# Findings: publish-frontmatter-image

**日時:** 2026-07-15
**対象:** plans/publish-frontmatter-image/

## 操作の事実

- 既存記事の frontmatter に画像パスを追加して更新した
- 本文＋frontmatter 両方に画像がある記事でも試した
- 確認方法: (1) 公開後のページで OGP 画像が表示されるか、(2) 公開先ディレクトリ/リモートに画像ファイルが送られているか、の両方を見た

## 利用仮説の照合

- 利用仮説（test-tree.md）: 「og_image に画像パスを指定した記事を公開すると、その画像がリモートに送られ、公開ページで OGP 画像が表示される」「本文とfrontmatterの画像が両方とも公開対象に含まれる」
- 実際: 期待どおりだった。OGP 画像表示・公開先へのファイル送信ともに想定した挙動になった

## 印象（ユーザーの言葉）

- 「特になし（想定どおりだった）」— 違和感・気になった点は挙げられなかった

## 成果物レビュー

語彙（plans/publish-frontmatter-image/dictionary.json）:
- 問題なし。`公開対象`・`画像参照抽出器`・`公開対象コレクター` の再定義（wip.status: redefine）は「本文のみ」→「記事由来のすべての画像参照（本文・frontmatter）」への拡張を正しく反映している
- 新規概念（`フロントマター画像参照抽出器`・`画像パス判定器`・`フロントマター解析器`）はいずれも `src` を持ち、関係リンクに孤立ノードなし（dict-search で参照整合を確認済み）

ソリューション構造（test-tree.md と現在の describe() 構造の比較）:
- `tests/editor/imageReference.test.js` は test-tree.md のツリーとほぼ一致した describe 構造になっている（公開対象コレクター → 画像参照抽出器 / フロントマター画像参照抽出器 → 画像パス判定器）。分解・合成の見直しは発生していない
- 差分: `tests/editor/publish.test.js` 内に、本プラン以前から存在する旧 describe ブロック「公開対象コレクターは現ファイルと参照画像を収集できる」「画像参照抽出器はMarkdownから画像パスを抽出できる」（77〜102行目）が残存している。これは `collectTarget` / `extractImageReferences` の**本文Markdown記法のみ**を前提にした旧語彙のテストで、辞書上は `公開対象コレクター`・`画像参照抽出器` とも再定義済みにもかかわらず、テスト側の名称・スコープは旧定義のままになっている。カバレッジは `imageReference.test.js` 側で拡張後の挙動として包含されており、内容的には重複

観察メモ（observations.md）:
- フロントマター解析器は新規実装ではなく `lib/pageData.js` の `parseMetaData` を named export 昇格して再利用した（重複実装を回避）
- 本文画像参照の定義はレビュー中に img タグを含む形へ拡張された（実装前に仕様の穴を塞げた）
- 公開対象コレクターで本文とfrontmatterが同一画像を参照するケースに気づき、Setによる重複排除を追加（テストツリーに明示ノードなし・ルートテストの範囲内の変更）

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: publish.test.js に旧語彙のテストブロックが未整理のまま残存
**戻し先:** 同一プラン（クローズ時のテスト二層化ステップで対応）
**出どころ:** 成果物レビュー
**解釈・原因の仮説:** `公開対象コレクター`・`画像参照抽出器` の再定義（本文のみ→本文+frontmatter）が `imageReference.test.js` という新規ファイルへの追加として実装されたため、`publish.test.js` 側の旧定義ベースの describe ブロックが更新対象から漏れた。テストツリーの構築が新規ファイル起点だったため、既存ファイル内の重複コードに気づきにくかったと考えられる
**推奨される対応:** クローズ時のテスト二層化で、`publish.test.js` の77〜102行目（旧 `公開対象コレクター`/`画像参照抽出器` ブロック）を畳む対象に含める。同等カバレッジは `imageReference.test.js` 側の昇格テストで維持される
**対応済み:** `publish.test.js` から旧ブロックと未使用importを削除（4テスト分）。同等カバレッジは `imageReference.test.js` に残る。`npm test` 338テスト全パス確認済み

### F-02: editor ハンドラーは静的依存グラフで実行時登録のため追跡できない（孤立ノードチェックでの既知の限界）
**戻し先:** tdd-skills（孤立ノードチェックの entry_points 定義に `packages/*/server/**` を加える検討）
**出どころ:** observations.md（tdd-run 手順7.5、本プランで確認）
**解釈・原因の仮説:** `packages/editor/server/publish.js` はサーバーディレクトリ機構による実行時登録のため、静的グラフの entry_points（`bin/*`, `lib/server/**`）から到達しない。今回は受け入れテスト（HTTP経由）で実到達を確認できたため実害はなかったが、editor コンテキストのハンドラー全般に共通する構造
**推奨される対応:** 今回は対応不要（受け入れテストで担保済み）。同種の見落としが繰り返されるようなら entry_points 定義の拡張を検討

### F-03: frontmatterImageReferenceExtractor.js が js/ ディレクトリに配置されているが実際は Node 専用
**戻し先:** 新規plan（将来クライアント側で画像パス判定を使う要求が出た場合のリファクタ）
**出どころ:** observations.md（tdd-run 手順7.5、本プランで確認）
**解釈・原因の仮説:** scaffold の慣例（フロントエンド関連は js/ に置く）に従ったが、`lib/pageData.js`（Node専用）に依存するためサーバー専用モジュールになっている。現状ブラウザから読み込む箇所はなく実害なし
**推奨される対応:** 今回は対応不要。将来ブラウザ側で画像パス判定が必要になった場合のみ、解析器の注入などで分離を検討
