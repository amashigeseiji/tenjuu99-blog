# Observations: publish-frontmatter-image

**日時:** 2026-07-15

## 実装中の気づき

- ツリーの組み換えは発生しなかった。問題定義の時点で既存語彙（公開対象コレクター・画像参照抽出器）の再定義が申し送りされており、ツリーはその延長で素直に構成できた
- フロントマター解析器は新規実装ではなく `lib/pageData.js` の内部関数（`parseMetaData` + `FRONTMATTER_REGEXP`）の named export 昇格で済んだ。既存実装の再利用をツリーのノードとして明示できたのは語彙検索（dict-search）が先にあったため
- scaffold.sh は plans/ 側の辞書を引かないため、新規概念の @vocab が英語名フォールバックになる。手動で日本語概念名に修正した（scaffold.sh が plans/*/dictionary.json も引けると手間が減る）
- 語彙提示の表を AskUserQuestion と同一ターンに置いたところユーザーに見えていなかった。表はターン末尾の最終メッセージで提示し、会話で承認を得る形に切り替えた
- 本文画像参照の定義はユーザーレビューで拡張された（Markdown 記法だけでなく img タグも含む。background-image 等スタイルシート経由は範囲外）。語彙定義の対話が仕様の穴（img タグ）を実装前に塞いだ
- 公開対象の収集で同一画像が本文と frontmatter の両方から参照されるケースに気づき、Set による重複排除を追加した（テストツリーには明示ノードなし・ルートテストの範囲内）

## スキルの使い方で想定と違ったこと

- 手順4の「利用仮説をユーザーと書く」は、ツリー承認と同じメッセージに含めて一括承認とした（対話コスト削減のため）
