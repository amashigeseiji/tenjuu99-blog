# Findings: error-visibility

**日時:** 2026-07-09
**対象:** plans/error-visibility/

## 操作の事実

- **起動失敗の再現:** blog.json の末尾にカンマを打って不正な JSON にし、.app を起動してそのディレクトリを選択した
  - 画面: 「サーバーの起動に失敗しました」の見出しの後にエラーログが表示された。「ログを表示」ボタンがあり、クリックすると Finder が開いた
  - ログ: `~/Library/Logs/tenjuu99-blog/` 以下に起動セッションごとのファイル。内容は `npm run server` 相当のエラーログ
- **起動後障害の再現:** 起動成功後、node プロセスを特定して kill で落とした
  - 画面: 「サーバーで問題が発生しました」と、通常のアプリログ（成功ログ）が表示された
- **正常系:** 正常なディレクトリを選んだ場合、エラー関連の表示は一切出ないことを確認した
- **ログの状態:** 確認時点でログファイルは5つ

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: SSG サーバー側のエラーメッセージが根本原因から遠い
**戻し先:** 新規 plan
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** blog.json のフォーマット不正という根本原因が、`helper.getPageDataByUrl` での null 参照という無関係な地点のエラーとして表面化する。設定読み込みの失敗を早期に検出・報告せず、後段まで null が伝播しているためと推測される。エラーの「可視化」（本プランのスコープ）は達成されたが、可視化された内容の「診断可能性」は SSG（Node）側の問題であり、本プランの範囲外（「個々のエラーの原因そのものの修正」は範囲外と明記済み）。
**推奨される対応:** 「エラーメッセージから根本原因に辿りつけない」を問題として `/tdd-problem` で新規プランを起こす（コンテキストは ssg-core / server 側）。

### F-02: 診断ログの保持上限が利用者から見えない
**戻し先:** なし（記録のみ）
**出どころ:** 利用インタビュー + 成果物レビュー
**解釈・原因の仮説:** 増大への懸念は実装上は解消済み（`maxSessions = 5` の世代管理）。ユーザーが5ファイルを見て懸念したのは、上限の存在が利用側から観察できないため。挙動としては仮説どおりで、伝達の問題に留まる。
**推奨される対応:** 現状は対応不要。将来、配布先ノンエンジニア向けの説明（README や表示文言でのひとこと）を検討する余地はある。

### F-03: tdd スキル群の JS 前提が Swift プランと摩擦
**戻し先:** tdd-skills
**出どころ:** observations.md
**解釈・原因の仮説:** scaffold.sh のコンテキスト分岐と depgraph（entry_points: bin/*, lib/server/**）が JS 前提のため、native-shell（Swift）プランでは手動スキャフォールドと孤立ノードチェックのスキップが必要だった。native 系プランが今後も続くなら都度のコストになる。
**推奨される対応:** tdd-skills リポジトリで scaffold.sh への native-shell 分岐追加と depgraph の多言語対応（または Swift 用の代替チェック手順の明文化）を検討する。

### F-04: problem.md の作業ディレクトリパス誤記
**戻し先:** tdd-skills（軽微）
**出どころ:** observations.md
**解釈・原因の仮説:** problem.md に存在しないパス `/Users/amashige/dev/tenjuu99-blog` が記録されていた（実際は `/Users/amashige/dev/private/tenjuu99-blog`）。tdd-problem がパスを検証せずに書き出している可能性。
**推奨される対応:** tdd-problem 側で作業ディレクトリの存在確認を入れる。

### F-05: ポート衝突時に失敗段階を誤判定しうる（既知の制約の記録）
**戻し先:** なし（記録のみ、将来の plan 候補）
**出どころ:** observations.md
**解釈・原因の仮説:** 他プロセスが 8000 番を専有していると readiness 検知が ready と誤判定し、実際は起動失敗なのに「起動後障害」と表示される。失敗段階の境界を「編集画面に到達したか」に置いているため。既存からある制約でスコープ外。
**推奨される対応:** 症状が実際に問題になったときに、ポートの主の識別を含めて新規プランで扱う。
