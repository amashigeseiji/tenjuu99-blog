# Findings: editor-sidebar-status

**日時:** 2026-06-29
**対象:** plans/editor-sidebar-status/

## 操作の事実

- 実装後、サイドバーを開いて編集済みファイルと新規ファイルにマークがついていることを確認
- 視覚的な改善のため CSS を書き換えた（マークの見た目を変更）
- 折り畳まれたディレクトリが編集中・新規ファイルを含むとき、ディレクトリノード自体を対応する色にする CSS を追加した（`data-status` 属性を利用）
- サイドバーの文字サイズ・折り返し周りの CSS を整理した

## 利用仮説の照合

- 利用仮説: `data-status` 属性が各ファイルノードに付与され、CSS で視覚的に区別できる → **期待側**
- 利用仮説: 新規作成後に別ファイルへ移動してもノードに `data-status="new"` が残る → **確認済み（マークが残っていた）**
- 利用仮説: すべてのファイルノードに `data-status` 属性がない → **外れ**（属性は正しく付与された）

## 印象（ユーザーの言葉）

- マーク自体は「悪くなかった」が、視覚的にもっとわかりやすくしたかったので CSS を書き換えた
- ディレクトリを閉じたときに中のファイルのステータスがわからなかった → CSS で対応した
- サイドバーの文字が大きく、ファイル名の折り返しで見辛い → CSS を整理した
- 元の問題（別ファイル移動後に未公開ファイルを見失う）は完全に解消された

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: デフォルト CSS が含まれていない

**戻し先:** 新規 plan
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** 実装は `data-status` 属性という CSS フックを提供したが、デフォルトのスタイルが不十分だったか存在しなかった。ユーザーが CSS を自分で書いて初めて「わかりやすい」状態になった。実装の責任範囲（HTMLの構造的正確さ）と視覚品質（デフォルト CSS）の分離が設計として正しかったかどうかは未検討。

**推奨される対応:** エディタのデフォルト CSS（`editor.html` か専用 CSS ファイル）に公開ステータスの基本スタイルを追加する plan を検討する。

### F-02: 折り畳みディレクトリへのステータス伝播は CSS の `:has()` で対応可能だった

**戻し先:** なし（情報として記録）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** 「ディレクトリを閉じたときに中のステータスがわからない」は懸念点だったが、`data-status` 属性を子孫に持つ `<details>` を CSS の `:has()` セレクタで選択することで対応できた。HTML 構造として `data-status` が正しい位置に付与されていたからこそ CSS だけで解決できた。

**推奨される対応:** 特になし。`:has()` による伝播をデフォルト CSS に含めることは F-01 の plan で検討できる。

### F-03: ファイル名末尾のマーク位置が折り返し問題を顕在化させた

**戻し先:** 新規 plan
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** サイドバーの文字サイズと折り返しの問題は実装前から存在していたが、ファイル名末尾にマーク（インライン要素）が追加されたことで折り返し時にマークが見えなくなるという問題が顕在化した。マーク位置の設計（末尾インライン vs 行頭・独立要素）と、サイドバーのテキストサイズ・折り返し設定が関係する。

**推奨される対応:** サイドバーの文字サイズ・折り返し・マーク配置を統合的に見直す plan を検討する。

### F-04: `ツリーレンダラー` 辞書エントリが実装と乖離している

**戻し先:** なし（辞書更新として対応）
**出どころ:** 成果物レビュー
**解釈・原因の仮説:** docs/dictionary.json の `ツリーレンダラー` エントリが「`renderTreeHtml(tree, activeFile)` 関数」と記述しているが、今回の実装で `statusMap` 引数が追加された。WIP 辞書にエントリがないため promote では更新されない。

**推奨される対応:** promote 後に docs/dictionary.json の `ツリーレンダラー` 定義を手動更新する。
