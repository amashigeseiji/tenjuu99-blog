# Observations: sync-operations

**日時:** 2026-07-11

## 実装中の気づき

### ツリー・語彙

- **除去・内容差分は describe の主語に現れなかった**。両概念はツリーの述語（「除去と取り込みを含む両向きで」）と他概念の定義の中で働く関係概念であり、単独のテストノード・実装装置を持たない。「静かに解決された」のではなく、構造を語るための語彙として機能している
- **削除する (Delete) の実装関数名は deleteArticle**。`delete` が JS の予約語のため、テスト→実装の名前対応に意図的な変換がある（deleteArticle.js の docblock に明記済み）
- **公開手段に refreshRemote（リモートの知識の更新）が実装中に追加された**。取り込みは fetch を前提にするため。WIP 辞書の定義には反映済み。promote 時の docs 反映に含めること

### スキルの使い方で想定と違ったこと

- **scaffold.sh の @vocab 逆引きが docs/dictionary.json しか見ない**。新概念は plans/<project>/dictionary.json にあるため、スタブの @vocab が英語名にフォールバックする。今回は scaffold と同形式のスタブを手書きして回避した。scaffold.sh が plans 辞書も見るようにする改善候補
- **依存グラフの孤立ノードチェックはエンドポイントファイルに適用できない**。packages/editor/server/*.js はサーバーがルートスキャンで動的にロードするため、静的グラフでは既存の publish.js も「被依存 0」になる。装置モジュール（pull.js・versionLineage.js・deleteArticle.js・changeReflector.js）はすべてエンドポイントから静的に到達可能なことを確認済み

### バグと検出経路

- **listRemoteFiles を構築時スナップショットで実装していた穴を、受け入れテストが検出した**。refreshRemote（fetch）後も古い一覧が返り、取り込みが「0件」になる。ユニットテストは「means を作り直してから一覧を見る」形で書いていたため素通りした。実際の利用順（構築 → refresh → 一覧）を通す受け入れテストが穴を突いた。ユニット側にも同順のテストを追加済み
- **UI 初期化完了前のボタンクリックが空振りする**。取り込みボタンは初期 disabled + ハンドラー登録時に有効化で対処。公開ボタンは既存からある挙動のためそのまま（既存挙動の維持）

### 範囲外として残したもの（将来の種）

- **分岐したまま公開すると push が拒否される**。取り込みが分岐記事を見送った後、その記事（や他の記事）を公開すると、手元の履歴がリモートより古いため push が失敗する（エラーは表示される）。これは本問題以前からの挙動で退行ではないが、「見送り→手元の版で上書き公開したい」という自然な次の欲求は競合解決の領域。problem.md の範囲外（競合解決の UI）と地続き
- **非公開は markdown のみをリモートから除去し、参照画像は残す**。画像は複数記事から共有されうるため安全側に倒した
- **手元で削除した公開済み記事を「明示的に取り込み直す（復元する）」操作は無い**。一括取り込みは削除の意図を守って見送る（理由は表示される）。復元が必要なら将来の単一ファイル取り込み操作の種
- **取り込みの対象範囲は src_dir 全体**（記事・画像・テンプレート）。記事だけに絞る選択肢もあったが、別マシンで足した画像が欠けると記事が壊れるため広めに取った

### 詰まりポイント

- 5回連続失敗のような詰まりは無し。受け入れテストのフレーク（エディタ初期化とクリックの競合、ダイアログハンドラーの二重登録）はテスト側の待ち方の問題で、publish-abstraction スペックの流儀（クリック前に初期ステータスを待つ）に合わせて解消
