## 問題

執筆者は、公開をやめる・記事を消す・リモートの内容を手元に取り込む、という操作をしたい。しかし現状のシステムに存在する同期操作は、実は一つだけである: **エディタで開いているファイルを、ローカルからリモートへ、追加・更新の向きで反映する**（Publish / Update）。やりたい3つの操作は、いずれもこの一文のどこかの制約を外すことを要求する。非公開は「追加・更新」ではなく*除去*の向きの反映であり、削除は*ローカルからの除去とその伝播*であり、取り込みは*リモート→ローカルという逆方向*である。

とりわけ削除は、現状のインターフェースの限界を最も純粋な形で露呈させる。公開処理はエディタで開いているファイルに対してしか行えないが、削除とは「開くものが無くなる」操作である。ローカルでファイルを消してしまうと、その削除をリモートへ反映する手段を失う。

### やりたいこと

- **非公開にする**: 一度公開した記事を、サイトから取り下げられる。原稿は手元に残り、再公開できる（語彙に「将来実装」として定義済み）
- **削除する**: 記事をローカルのファイルシステムから削除でき、その削除がリモートにも反映される（リモートに残骸が残り続けない）
- **取り込む（pull する）**: リモートの内容をローカルに取り込める。別のマシンで公開した記事を、手元でも最新の状態にできる

### 障害・制約

- **操作の主体が「開いているファイル」に縛られている**: 公開処理はエディタで開いているファイルにしか行えない。削除するとファイルは開けなくなり、リモートへ反映する手段を失う。したがって「非公開にする → 削除する」という順序を強制するか、ファイルを開かなくてもローカル/リモートの同期が可能になっている必要がある。取り込みも同根である——リモートで増えた・変わったファイルは手元で「開く」ことができないので、ファイル単位の操作では原理的に取り込めない。削除の伝播と取り込みは、ファイルに紐づかない同期という同じ要求を両側から指している
- **除去の向きが語彙にない**: 現行の反映（`公開する`・`更新する`・`変更反映器`）は追加・更新のみを扱う。`非公開にする`・`削除する` は `公開ステータス` への遷移として種があるのみで、リモートから*取り除く*反映は概念としても実装としても存在しない
- **リモート→ローカル方向の概念が存在しない**: `リモート状態` はリモート内容の読み取り専用の参照であり、内容の取り込みではない。現語彙にある方向はローカル→リモートのみ
- **取り込みの安全性と「先/後」**: 取り込みが手元の執筆内容を上書きしてはならない。しかし「手元とリモートのどちらが進んでいるか」という向きのある判定は、版の連なり（履歴・共通の祖先の知識）があって初めて言える。内容の比較だけで言えるのは「一致・不一致・欠如」までである。git 手段はこの連なりを自然に持つが、手段の抽象がこの能力をどこまで要求するかは決まっていない
- **不在の曖昧性**: 取り込みが入るとリモートも変更源になるため、「手元に無い」が「削除した（伝播待ち）」なのか「他所で作られた（取り込み待ち）」なのか、ファイルの有無だけでは区別できない。削除の伝播と取り込みは、互いに他方のノイズになりうる

### 観察された症状

- 公開をやめたい・記事を消したいという操作の置き場所が UI に存在しない。ローカルでファイルを消しても、リモート（＝サイト）には残り続ける
- 別のマシンで公開した記事を手元に取り込むには、手動の git pull が必要。git を理解している人にしか実行できない

## 範囲外

- **下書きの退避・別マシンでの下書き作業**: `index: false, noindex: true` で公開する運用（サイト内リンクが無いため bot に辿られず、URL を知らない限り見つからない）で実質的に回避できているため、後回しにする。将来の問題の種として申し送りに記録
- **複数人での共有と編集セッション（編集権ロック）**: 競合を予防する中心概念として立てる方向はユーザーと合意済みだが、本問題では扱わない。1人が複数台を順に使う範囲では競合は稀である
- **ライフサイクルのラベル体系**: 下書き・編集中・公開済みといった状態の読みの体系化。本問題は基盤となる同期操作の欠けを埋めるところまで
- リアルタイム共同編集、ユーザー認証・アカウント管理、競合解決の UI（マージツール）
- CI/CD パイプラインの設定・変更

## 解決したと言える状態（問題領域の言葉で）

- 公開済みの記事を非公開に戻せる。原稿は手元に残り、再公開できる
- 記事を削除でき、その削除がリモートにも反映される。リモートに残骸が残り続けない
- リモートの内容を手元に取り込め、別のマシンで公開した記事の続きを手元で書ける
- 取り込み・削除の伝播によって、執筆した内容が失われない
- これらの操作が、git の言葉（pull・merge・コンフリクト等）を使わずに実行できる
- git 手段での既存の公開運用が壊れていない（publish-abstraction の達成事項を退行させない）

## 技術的背景（調査結果）

- `ローカル`・`リモート` は publishing/application のドメイン概念として昇格済み。公開手段の抽象（`公開手段`・`公開手段解決器`・`git公開手段`）も確立済みで、本問題の新しい操作もこの建て付け（意味は手段から独立、git は一実現）に載る
- **背景モデル（本問題に先立つ対話で得た理解）**: ページのライフサイクルは、独立した状態機械ではなく、場所（手元/リモート/サイト）間の差分の**読み**として発現する。現行 `公開ステータス`（未公開/更新あり/公開済み）はまさにその読みであり、`非公開にする` の「一度公開したものと一度も公開していないものを区別しない」という定義は、差分から導出できない区別を意図的に捨てている証拠である。本問題の3操作は、この読みの基盤となる同期操作の欠け（除去の向き・逆方向）を埋めるものにあたる
- 作者の git 実践で使われていた操作は push / pull のみ（ブランチ操作はほぼ無し）。この運用が git から暗黙に得ていた性質は、分岐の検出（リモートが進んでいたら push を拒否）、履歴（向きの判定・復元・不在の曖昧性の解消）、原子性であり、同期手段の抽象への要求候補になる
- エディタの公開系エンドポイント（`/publish`・`/get_sidebar`・`/publication-status`）は開いているファイルまたはファイル一覧を対象とし、ファイルの不在を扱う経路が無い

## /tdd-run への申し送り

- **語彙定義の種**: 同期の方向（届ける/取り込む——push/pull の手段非依存の言い直し）。除去の反映（非公開＝リモートからの除去、削除＝ローカルからの除去とその伝播）。ファイルに紐づかない同期（開いているファイル単位ではない操作）。差分の階層——内容比較で言えるもの（一致・不一致・欠如）と、版の連なりがあって初めて言えるもの（先行・後行・分岐）の区別
- **開いている点（削除の伝播の設計）**: 「非公開にする → 削除する」の順序を強制するか、ファイル非依存の同期を導入するか。どちらも障害・制約に記した同じ制約の解であり、solution domain で選ぶ
- **開いている点（向きの担保）**: 取り込みの安全性に必要な「先/後」の判定を、(a) 手段に要求する（git は自然に満たす。FTP はマニフェスト等の追加装置が要る）、(b) 手段から独立にシステム自身が版の連なりを保持する、(c) 要求せず「不一致」までで正直に退化させる——のどれをどの水準で採るか
- **開いている点（不在の曖昧性）**: 「削除した（伝播待ち）」と「他所で作られた（取り込み待ち）」の区別を、履歴に頼るか、削除の記録（tombstone）を持つか、中間状態を作らない操作設計で回避するか
- **将来の問題の種（下書きと到達可能性）**: `noindex` 公開の運用は、「公開」がバイナリではなく到達可能性のグラデーションを持つことを示している（「公開されているが見つからない」状態がある）。下書きが「場所」ではなく「公開物の属性」で実現されている例であり、将来ライフサイクル/下書きの問題を起こすときの出発点になる
- **将来の問題の種（編集セッション）**: 複数人共有に進むとき、「真実の源泉は場所ではなくバトンである」——編集セッションを開いた手元が、その間だけ真実を持ち、ロックにより同時にひとつ——という方向を合意済み
- **利用検証の基準**: 3機能とも作者自身が現に必要としているものであり、利用インタビューは作者の実利用で行える
