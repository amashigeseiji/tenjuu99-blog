# テストツリー: sync-operations

**作成:** 2026-07-11

## できるのツリー

```
同期 はローカルとリモートの内容を、除去と取り込みを含む両向きで揃えられる [publishing]
├── 公開手段 は反映に加えて、リモートからの除去・取り込み・版の連なりの参照を提供できる [publishing]
│   └── git公開手段 は除去・取り込み・版の連なりの参照を実現できる [publishing]
├── 非公開にする は公開済みの記事をリモートから取り除き、原稿を手元に残せる [editor]
├── 削除する は未公開の記事をローカルから取り除ける [editor]
├── 取り込む はリモートの内容を、手元の執筆内容を失わずにローカルへ反映できる [publishing]
│   └── 版の連なり は記事ごとに先行・後行・分岐と、手元に無い記事の事情（消された・他所で作られた）を判定できる [publishing]
├── ファイルステータスコレクター は手元に無いがリモートにあるファイルも含めて記事の状態を収集できる [editor]
└── 公開可否判定器 は記事の状態ごとに行える操作（公開・非公開・削除・取り込み）を見分けられる [editor]
```

エントリポイント: `packages/editor/server/*.js`（エンドポイント群。新設 /unpublish・/delete・/pull と既存 /get_sidebar の拡張） → 同期

セミラティス: **版の連なり** は「取り込む」の安全性判定と「ファイルステータスコレクター」の不在の事情の区別、両方から参照される共有ノード。

### 設計上の合意（problem.md の開いている点への回答）

- **削除の伝播**: 存在しない。除去は一操作一場所（非公開＝リモートから、削除＝ローカルから）。削除は未公開のファイルにのみ提供され、公開済みの記事は非公開→削除の2段になる（UIが誘導する）
- **向きの担保**: 手段に要求する（申し送りの選択肢 (a)）。git は版の連なりを自然に持つ。満たさない手段は内容差分の「不一致」までで正直に退化する
- **不在の曖昧性**: 版の連なりで区別する。自動伝播を持たないため、最終判断は同期ビュー（サイドバー拡張）を見た執筆者が行う

## 利用仮説

- 使ったらこうなるはず: 作者が実運用で「公開をやめる」「記事を消す」「別マシンで書いた記事を取り込む」を、git コマンドを一度も打たずにエディタ内で完了できる
- こうなったら外れ: 取り込みで執筆内容が巻き戻る／削除した記事が復活する／UI に git の言葉（pull・merge・コンフリクト）が現れる／既存の公開運用（Publish/Update）が退行する
