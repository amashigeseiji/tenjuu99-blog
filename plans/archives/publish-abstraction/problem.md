## 問題

このブログツールをアプリとして配布する試みの中で、Node.js などの依存はパッケージ内に
閉じ込めることができた。最後に残った外部依存が git である。利用者の環境に git が
インストールされていることを要求せずに配布できる状況を目指したいが、その手前に
より根本的な障害がある。

「公開する」という行為の意味そのものが、現在は git 運用の言葉でしか定義されていない。
「公開済みとは何か」はリモートリポジトリとの一致、「公開するとは何をすることか」は
commit + push、「どうやってサイトに反映されるか」は push をトリガーとする CI/CD の
デプロイ——という具合である。このため、FTP/SFTP のような別のプロトコルであれ、
JS による git 実装の内蔵であれ、手段を替える議論を始めようとすると、そのたびに
「そもそも公開とは何をすることか」から考え直すことになる。

### やりたいこと

- 「公開する」を、特定の手段（git 運用）の言葉を使わずに定義できる状態にしたい。
  手段（git / FTP / SFTP / JS git 実装など）は、その定義の実現方法の一つとして
  位置づけられる
- その定義の上で、後続の問題（git 非依存での配布、個別プロトコルへの対応）を
  「公開とは何か」を再定義することなく立てられるようにしたい

### 障害・制約

- **意味の癒着**: 語彙（docs/dictionary.json）と実装の双方で、「公開する」「公開済み」
  「デプロイ」の定義文自体に git 運用（リモートリポジトリ、commit/push、CI/CD）が
  埋め込まれている
- **届けるものが手段によって変わる**: git 運用では原稿（Markdown ソース）を送れば
  CI/CD がビルドして配信するが、ビルドをしてくれない届け先（FTP サーバーなど）では
  成果物を届ける必要があるかもしれない。「何を届ければ公開したことになるのか」が
  手段ごとに異なりうるため、公開の定義はこの違いを跨いで成立しなければならない
- **公開済みかどうかを知る手がかりが手段ごとに異なる**: 公開ステータス
  （未公開・更新あり・公開済み）の判定は「リモートの現在の内容を参照できること」に
  依存している。git はこれが得意だが、履歴や差分の概念を持たない届け先では参照の
  しやすさが大きく異なる
- **既存運用との両立**: git ベースの公開（commit + push → CI/CD デプロイ）は
  今後も選択肢の一つとして対等に残す。既存の運用を壊す形での再定義はできない

### 観察された症状

- FTP や SFTP など、git 以外のプロトコルで公開できない
- アプリ配布の観点では、利用者に git のインストールを要求してしまう

いずれも表層であり、背後にあるのは「公開の意味が特定の手段と癒着している」という問題である。

## 範囲外

- 個別プロトコル（FTP/SFTP など）への実際の対応 — 本問題の解決を前提とする後続問題
- JS による git 実装を内蔵するかどうかの判断 — 手段選定であり後続問題
- git なしで配布可能なアプリパッケージの成立 — 配布パッケージング側の問題
- CI/CD パイプラインの設定・変更 — 従来どおりエディタの範囲外

## 解決したと言える状態（問題領域の言葉で）

- 「公開する」「公開済み」「更新あり」といった概念が、git の言葉を使わずに
  定義されている
- git による公開は、その定義の一つの実現として位置づけ直されており、既存の
  git 運用は従来どおり成立している
- 新しい公開手段への対応を検討するとき、「公開とは何か」に立ち戻る必要がなく、
  その手段がこの定義をどう実現するかだけを考えればよい状態になっている

## 技術的背景（調査結果）

- git コマンドの実行は `packages/editor/server/publish.js` 内の2つのファクトリ
  （`createGitPublishedState`, `createGitPublishActions`）に閉じている。
  ドメインロジック（`handlePublish`）や公開ステータス判定
  （`publicationStatus.js`, `sidebarStatusCollector.js`）は注入された抽象越しに
  動作しており、コード上の git 依存はすでに局所的
- 一方、docs/dictionary.json の語彙定義は git 前提が埋め込まれている:
  - 「デプロイ」=「リモートリポジトリへの送信をトリガーに…」
  - 「公開済み状態」の定義文が git（remote HEAD, `existsInRemote`,
    `diffFromRemote`）を参照
  - native-shell コンテキストの記述にも「公開する操作（Publish/git操作）」の表現
- つまり実装よりも概念・語彙のレベルで癒着が残っており、これが「手段を替える
  議論のたびに公開の意味から考え直す」状況を生んでいる

## /tdd-run への申し送り

- 語彙定義の種（既存辞書で語れなかった概念）:
  - 「公開手段」（仮名）— git / FTP / SFTP / JS git 実装などを総称する、公開の
    実現方法の概念。現状の辞書には存在しない
  - 「公開物」（仮名）— 公開の際に届けられるもの。原稿（ソース）か成果物
    （ビルド済みサイト）かが手段によって異なる。現状は暗黙に「原稿」に固定
- 既存語彙の改訂候補: 「デプロイ」「公開する」「公開済み状態」の定義文から
  git 前提の表現を分離する必要が出る見込み（/tdd-vocab plan で扱う）
- 開いている点:
  - 「公開した」と言える瞬間はどこか。git + CI/CD では push 完了で公開操作は
    完了扱いだが、サイトへの実反映は非同期に起きる。手段によっては送信完了と
    サイト反映が一致する。この揺れを定義がどう吸収するかは未確定
  - 公開ステータス判定が「リモートの内容を参照できること」を要求する度合い。
    参照が困難な手段では判定の意味自体が変わりうる
- 後続問題（FTP/SFTP 対応、git 非依存配布など）を立てる際は、本問題を
  `前提とする` 関係として宣言すること
