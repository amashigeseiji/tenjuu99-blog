## US-01: 手段を意識せずに記事を公開できる

**ロール:** ブログ執筆者（エディタ利用者）
**目的:** 公開操作を行い、記事を公開済みにしたい
**価値:** どの公開手段が構成されていても、同じ操作・同じ知覚で公開が完結する

### シナリオ 1: 未公開の記事を公開する（git 手段・既存運用の維持）

```
Given git を公開手段とするブログプロジェクトに、公開ステータスが「未公開」の記事がある
When 執筆者が公開操作を行う
Then 公開が成功したことがブラウザ上で知覚でき、記事の公開ステータスは「公開済み」になり、
     従来どおりサイトへの反映フローが起動する
```

### シナリオ 2: 公開済みの記事を編集して更新する（git 手段・既存運用の維持）

```
Given git を公開手段とするブログプロジェクトに、公開ステータスが「更新あり」の記事がある
When 執筆者が公開操作を行う
Then 更新が成功したことがブラウザ上で知覚でき、記事の公開ステータスは「公開済み」になる
```

### シナリオ 3: 公開済みの状態が参照できないとき

```
Given 公開手段がリモートの現在の内容を参照できない状態にある
When 執筆者が公開操作を行う
Then 公開は実行されず、参照できない旨のフィードバックがブラウザ上で知覚できる
```

## US-02: 「公開する」を手段の言葉なしで語れる

**ロール:** ツールの開発者（公開手段の追加・保守をする人）
**目的:** 公開に関わる概念を、特定の手段（git 等）から独立に理解・参照したい
**価値:** 手段を替える議論のたびに「公開とは何か」から考え直す必要がなくなる

### シナリオ 1: 公開の語彙が手段の言葉を含まない

```
Given 公開に関する語彙定義（公開する・更新する・公開済み・公開ステータス・デプロイ等）がある
When 開発者がその定義を読む
Then 概念の定義文に特定の手段（git、コミット、プッシュ、リモートリポジトリ等）の言葉が
     現れず、手段はその定義の実現方法の一つとして別に位置づけられている
```

### シナリオ 2: git による公開が「一つの実現」として位置づいている

```
Given 手段の言葉を含まない公開の定義がある
When 開発者が既存の git ベースの公開の仕組みを確認する
Then git による公開は、その定義の各要素（公開済みかどうかの判定・公開の実行・
     届けるもの）をどう実現しているかとして説明でき、既存の運用は従来どおり動作している
```

### シナリオ 3: 新しい公開手段の検討が定義の再訪を要求しない

```
Given 手段の言葉を含まない公開の定義がある
When 開発者が新しい公開手段（例: FTP/SFTP）への対応を検討する
Then 検討すべきことは「その手段が定義の各要素をどう実現するか」に限られ、
     「公開とは何か」の再定義は必要にならない
```
