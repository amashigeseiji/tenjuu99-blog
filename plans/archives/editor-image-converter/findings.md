## 発見（解釈・原因の仮説、戻し先つき）

### F-01: acceptance テストが Playwright を使いながらブラウザ操作をしていなかった

**戻し先:** 次回 acceptance テストを書くサイクル
**出どころ:** observations.md
**解釈・原因の仮説:** Playwright は「ブラウザ操作のないテスト」も実行できるため、構文エラーにならず見落とされやすい。ユーザーストーリーの「編集画面からアップロードする」は HTTP エンドポイントを通じた検証が適切だが、関数直呼びで代替されていた。`webServer` が不要に見えたため削除を検討したが、将来の e2e テストのために必要な設定だった。
**推奨される対応:** acceptance テストを書く際は、ユーザーストーリーの「When」句が UI 操作やサーバー経由の操作を指している場合、`page.goto()` + `request.post()` 等で実際のエンドポイントを通じて検証する。関数直呼びは実装の詳細テストに留める。

---

### F-02: `lib → packages` のクロスパッケージ依存

**戻し先:** 新規 plan（リファクタリング）
**出どころ:** observations.md
**解釈・原因の仮説:** `lib/distribute.js` が `packages/editor/server/image_upload.js` の `createConverter` をインポートしている。SSGコアが editor パッケージに依存する形になっており、依存方向が逆転している。今回のスコープでは修正せず据え置いた。
**推奨される対応:** `createConverter` を `lib/` に移動するリファクタリング。ただし editor パッケージの他の処理との兼ね合いを調査してから判断する。

---

### F-03: ビルド画像配布器（generate パス）が実際には未検証のまま

**戻し先:** 任意（次回利用時に確認）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** テストは通っているが、実際の `generate` コマンド実行での出力確認が行われていない。アップロードパス（editor）は動作確認済みだが、ビルドパス（ssg-core）は接地していない。
**推奨される対応:** 次回 `generate` を実行する機会に `dist/image/` の拡張子・サイズを確認する。問題があれば新規 plan に。
