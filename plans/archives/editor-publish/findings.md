# Findings: editor-publish

**日時:** 2026-06-27（第2サイクル — F-04/F-05 修正後）
**対象:** plans/editor-publish/

---

## 操作の事実

- 新規ファイルを作成し、Publish ボタンを押した
- commit が実行された
- `.git/hooks/pre-push`（`ALLOW_PUSH=1` でバイパス）が push を抑制し、エラーで止まった
- エラー表示には git のメッセージが含まれていた

---

## 利用仮説の照合

| 仮説 | 結果 |
|------|------|
| Publish ボタンでターミナル操作なしに commit/push が実行され、結果が画面に表示される | 期待側（push は開発環境の hook でブロックされたが commit は実行、意図通りの挙動） |
| push 対象に現在ファイル＋参照画像が含まれる | 未確認（問題ないと判断） |
| 公開ステータスがファイルごとに表示される | 遷移は未確認（hook 抑制のため）だが問題ないと判断 |
| 保存操作がサーバーリロードを引き起こし直後の publish が失敗する（外れ側） | 外れ側にならず（F-04 対応済み） |
| 「Failed to fetch」がそのまま表示される（外れ側） | 未確認だが変換済み（F-05 対応済み） |

---

## 印象（ユーザーの言葉）

- エラー表示に git メッセージが含まれていた → 「まあ良いとおもいます」
- 画面上部右側のボタン群（公開状態|preview|save|publish|戻る）を整理したい
  - preview ボタン：自動プレビューがすでにあるので不要
  - save ボタン：まだ必要だが、自動保存があって然るべき
- 新規作成 UI が使いにくい
  - セレクトボックスの最上部に「新規作成する」があり分かりづらく、ファイル数が多いと遠い
  - 新規作成を選んだ後、空のインプットフォームにファイル名を入力するという規則が分かりづらい
  - 独立した「新規作成」ボタンがあるべき
  - 新規作成でファイル名が入力されたらファイルが作成（save）されるが公開はされない
  - 新規作成時にテンプレート選択ができると良い

---

## 発見（解釈・原因の仮説）

### F-01: エラーメッセージに git の生テキストが含まれる
**戻し先:** 新規 plan（エディタ UI 整理）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** F-05 対応はネットワーク到達不能を「サーバーに接続できませんでした」に変換したが、git 操作失敗時のメッセージはそのまま渡っている可能性がある。ユーザーは許容しているため優先度は低い。
**推奨される対応:** git エラーの生テキストをユーザー向けの言葉に変換するか、詳細はコンソールに送りつつ画面には「公開に失敗しました」等を表示する。

### F-02: preview / save ボタンが機能的に冗長になった
**戻し先:** 新規 plan（エディタ UI 整理）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** リアルタイムプレビュー（editor-realtime-preview）が導入されたことで preview ボタンは役割を失った。save については自動保存が実装されれば同様に整理できる。フィーチャーを積み上げるたびにボタンが増え、UI の認知負荷が上がっている。
**推奨される対応:** preview ボタンを削除。save は自動保存の実装と合わせて整理する。

### F-03: 新規作成 UI の発見性と操作規則が分かりづらい
**戻し先:** 新規 plan（新規作成 UI 改善）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** 「新規作成する」がセレクトボックス内に埋め込まれており、ファイル選択と新規作成が同一 UI に混在している。空のインプットフォームへの入力という規則も初見では分かりにくい。新規作成はファイル選択とは独立したフローとして切り出すべき。
**推奨される対応:** 独立した「新規作成」ボタンを設置。ボタン押下でファイル名入力・テンプレート選択のフォームを表示し、確定でファイル作成（save のみ、公開はしない）する。
