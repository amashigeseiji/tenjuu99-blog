# Findings: content-root-switching

**日時:** 2026-07-08
**対象:** plans/content-root-switching/

## 操作の事実

- ユーザーは実アプリを起動して使用した。起動後、「ファイル」メニューから blog.json のあるディレクトリへ移動（切り替え）すると、アプリはそのプロジェクトで動いた
- 複数のプロジェクト間を何度か行き来し、いずれも問題なく動作した
- 切り替え後にアプリを再起動したところ、前回開いていたプロジェクトでダイアログなしに立ち上がった
- （tdd-run 中の検証）シナリオ3（キャンセル→現状維持）は AppleScript による Cancel クリックで検証済み

## 利用仮説の照合

- 仮説「メニューから blog.json を持つプロジェクト B を指定すると B の内容で使える状態になる」（US-01 シナリオ1）→ **期待側**
- 仮説「切り替え後に再起動するとダイアログなしで B のまま立ち上がる」（US-01 シナリオ2）→ **期待側**
- 仮説「フォルダを選ばずにやめれば現状維持」（US-01 シナリオ3）→ **期待側**（tdd-run 中の AppleScript 検証による。実利用インタビューでは未言及）
- 「永続化ストアの手動操作などアプリ外部の知識を必要としない」→ **期待側**（メニュー操作のみで完結）
- 外れ側の兆候（表示とサーバーの不一致、キャンセルでの記憶消失、blog.json 無しフォルダでのアプリ終了、旧サーバー残留）はいずれも観察されていない

## 印象（ユーザーの言葉）

- 「何度か行ききして、問題なく動いています」
- 「メニューが『ファイル』よりは『プロジェクト』とかのほうがわかりやすいかも、くらいです。このままでも構いません」

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: メニュー名「ファイル」は切り替えの単位語「プロジェクト」とずれている
**戻し先:** 新規 plan（軽微。次に native-shell を触るプランへの同乗でも可。ユーザーは「このままでも構わない」と明言しており、対応は必須ではない）
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** 辞書は切り替えの単位を「プロジェクト」と定義したが、メニューの見出しは AppKit 慣習の「ファイル」のまま。ユーザーが自発的に「プロジェクト」という語で言い直したことは、辞書の語彙選択が利用実感と合っている傍証であり、UI の表示語だけが語彙ネットワークから外れている状態
**推奨される対応:** 対応する場合はメニュー見出し（またはメニュー項目名）を「プロジェクト」系の語に寄せる。単独プランを立てるほどの規模ではない

### F-02: 依存グラフが Swift と scripts/ を収録しておらず、孤立ノードチェックの枠外
**戻し先:** tdd-skills（手順7.5 の適用範囲の問題）
**出どころ:** observations.md
**解釈・原因の仮説:** depgraph は JS のみ（entry_points は bin/* と lib/server/**）を収録しており、native/ 配下の Swift と scripts/ 配下の JS は構成把握の対象外。native-shell 系プランが続く限り、このギャップは毎回観察され続ける
**推奨される対応:** depgraph の entry_points 拡張（scripts/ 追加）か、Swift を枠外と明示する運用の確立

### F-03: 切り替え機能により node_modules 破壊問題が顕在化しやすくなった
**戻し先:** 既存 plan（plans/node-modules-destruction、新規起票は不要）
**出どころ:** observations.md / problem.md の「他の問題との関係」
**解釈・原因の仮説:** 切り替え時の `linkAppNodeModulesIntoContentRoot` は既存実装の再利用であり、実体 node_modules を持つプロジェクトへ切り替えると破壊が起きる状況は従来どおり。切り替えが容易になったぶん、遭遇確率は上がっている
**推奨される対応:** plans/node-modules-destruction の優先度を上げる判断材料とする

### 記録: 副産物として .app 再ビルド不能バグと絶対パスリンクの潜在配布バグを修正済み
**戻し先:** なし（解決済み）
**出どころ:** observations.md
再ビルド時の `ERR_FS_CP_EINVAL`（self-reference リンク × 既存 dest）を bundleWriter.js のクリーン書き出し + verbatimSymlinks で修正し、再現テストを tests/app-bundle/bundleWriter.test.js に追加（既存の安定概念「バンドル書き出し器」の describe 配下）。旧 dist-app の .app 内絶対パスリンク（配布先でモジュール解決が壊れる潜在バグ）も再ビルドで相対リンクに置き換わった。
