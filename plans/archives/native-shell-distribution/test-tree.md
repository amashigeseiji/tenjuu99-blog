# テストツリー: native-shell-distribution

**作成:** 2026-07-03

## できるのツリー

```
配布物は Node未インストール環境で自己完結して起動し編集操作まで行える [app-bundle]
├── バンドルレイアウト解決器は .app内のNode実行ファイルとアプリコードエントリーの場所を導出できる [app-bundle]
│
├── コンテンツルート解決器は blog.jsonを含むフォルダの場所を、記憶済みでなければユーザーに選択させて確定できる [app-bundle]
│   └── （手動検証のみ）コンテンツルート選択ダイアログは ユーザーにフォルダ選択UIを提示できる [app-bundle]
│
├── 配布物組み立て器は Node本体・ネイティブ依存・ソースコード一式を.app構造に配置できる [app-bundle]
│   ├── 配置マッピング解決器は バンドルマニフェストをビルド成果物の実パスに解決できる [app-bundle]
│   └── バンドル書き出し器は 解決済みの対応表に従って実際にファイルをコピーし.appを組み立てられる [app-bundle]
│
└── （手動検証のみ）ネイティブアプリシェルは バンドルレイアウト解決器とコンテンツルート解決器の結果を使ってサーバーを起動できる [native-shell]
```

## 責務の境界（設計上の決定）

- `バンドルレイアウト解決器`は `.app` に同梱された読み取り専用リソース（Node実行ファイル・アプリコードエントリー）の場所のみを扱う。存在は同梱物として保証されているため「無かったら」を考える必要がない。
- `コンテンツルート解決器`は「`blog.json`を含むフォルダは既に存在する」ことを前提とする。選ばれたフォルダに`blog.json`が無い場合はエラーとして扱う（新規作成・自動生成はこの装置の責務に含めない。別途の関心事として保留）。
- `コンテンツルート記憶`（永続化）は独立ノードにしない。`ServerLifecycleBinding`が`Process`を直接ラップして専用ノードを持たないのと同様、コンテンツルート解決器に注入される配線の一部として扱う。

## ロード可能性チェック（Swift/AppKit・Node混在）

- `バンドルレイアウト解決器`: AppKit非依存の純粋ロジック → 単体テスト可能（`swift run CoreTests`）
- `コンテンツルート解決器`: 永続化ストア・選択ダイアログを関数として注入する設計にすることで、コアロジック自体はAppKit非依存にする → 単体テスト可能
- `コンテンツルート選択ダイアログ`: NSOpenPanel直結で切り出せる純粋ロジックがない → 手動検証のみ
- `配布物組み立て器` / `配置マッピング解決器` / `バンドル書き出し器`: Node.js側の純粋ロジック・fs操作。ブラウザ/GUI依存なし → 単体・統合テスト可能（`npm test`、tmpディレクトリを使用）
- `ネイティブアプリシェル`（AppDelegateの配線変更）: AppKit直結、ロード時に画面操作の副作用があるため切り出せる純粋ロジックがない → 手動検証のみ（既存の`native-mac-shell`と同じ扱い）

## 利用仮説

- 使ったらこうなるはず:
  - Node/sharpがインストールされていないクリーンなMacに`.app`を置いてダブルクリックすると、初回はコンテンツルート選択ダイアログが表示され、`blog.json`を含むフォルダを選ぶとサーバーが起動し編集画面が表示される（US-01）
  - 同じ`.app`を再度起動すると、記憶済みのコンテンツルートがそのまま使われ、ダイアログは表示されない
  - `.app`が内包するNode本体・sharpのネイティブバイナリ・ソースコード一式だけでサーバーが起動し、外部にインストールされた`node`コマンドやグローバルパッケージへの依存が発生しない（US-02）
- こうなったら外れ:
  - バンドルレイアウト解決器が誤ったパスを返し、Node実行ファイルが見つからずサーバー起動に失敗する
  - コンテンツルート解決器が`blog.json`の存在確認を誤り、存在しないフォルダを確定済みとして扱ってしまう
  - 配布物組み立て器がマニフェストの一部を反映し損ね、`.app`にNode本体・sharp・ソースコードのいずれかが欠落する
