# Findings: native-shell-distribution

**日時:** 2026-07-07
**対象:** plans/native-shell-distribution/

## 操作の事実

- 利用環境は開発者自身の Mac のみ。クリーンな（Node 未インストールの）Mac での検証は困難なため未実施。
- ビルド手順を教わって `scripts/app-bundle/build-local.js` でビルドし、`dist-app/tenjuu99-blog.app` を Finder からダブルクリックで起動した（ビルド手順自体は本人はもう覚えていない）。
- 初回起動時に `blog.json` のあるディレクトリの選択画面が表示され、選択後は問題なく立ち上がった。
- 2回目以降の起動では、ダイアログなしで前回のフォルダのまま立ち上がった。
- 別プロジェクト（`~/dev/nobi`）をコンテンツルートに選んだところ、既存の実体 `node_modules` がシンボリックリンクに置き換えられた（現時点で実害は出ていない）。
- コンテンツルートを選びなおすためにアプリ再起動を試したが、記憶済みフォルダのままで変わらなかった。
- エラー発生時は画面が真っ白になる。「サーバーを起動中です」の表示から真っ白になるパターンと、最初から真っ白のパターンがある。どこを見れば内容が分かるかはユーザーには分からない。

## 利用仮説の照合

- 仮説1（初回起動: ダイアログ → フォルダ選択 → サーバー起動 → 編集画面）: **期待側**（ただし自分の Mac 上での確認。クリーン Mac では未検証）
- 仮説2（再起動時は記憶済みコンテンツルートを使いダイアログなし）: **期待側**
- 仮説3（`.app` 内包物のみで動作し外部の node に依存しない = US-02）: **どちらとも言えない**（ユーザー自身の実利用では検証できていない。tdd-run 時の手動検証記録はあるが、クリーン環境での確認は次フェーズのまま）
- 外れ条件（誤ったパス導出・blog.json 確認の誤り・同梱物の欠落）: いずれも観測されず

## 印象（ユーザーの言葉）

- 「現状ではいくつか困っていることがあり」
  1. 「一度選択したフォルダを解除して選択しなおす術がない」→ ほどくと「この Mac 内に複数のプロジェクトがあるので、それぞれで試したい。アプリは単一だが、プロジェクトが複数ある」「メニューバーなどに『コンテンツフォルダを変更』があるのがよい」
  2. 「node_modules がシンボリックリンク上書きされるから、ローカルで既に開発中のものと同居できない」→ 事実確認: nobi をコンテンツルートにしたとき、既存の node_modules がリンクに置き換えられた。「いまは問題ないようです」
  3. 「エラーが発生したときに内容がまったくわからない」→ 「真っ白です。どこを見にいけばいいかはわかりません」
- ビルドについて: 「もうビルドの仕方を忘れましたが」（一度教われば実行はできたが、手順が定着していない）

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: コンテンツルートの切り替え手段がない（単一アプリ×複数プロジェクト）
**戻し先:** 新規 plan
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** 元の問題定義は「ノンエンジニアの知人に単一プロジェクトを渡す」ことを対象にしており、コンテンツルート解決器は「記憶済みなら常にそれを使う」設計で問題なかった。実際の利用者（開発者自身）は複数プロジェクトの切り替えを求めており、これは問題定義のスコープの外側で生まれた新しい要求。UserDefaults の記憶キーを手動で消す以外の切り替え手段が UI に存在しない。
**推奨される対応:** 「メニューバーに『コンテンツフォルダを変更…』」を軸とした新規 plan として /tdd-problem を起動する。F-02 と関心が重なる（プロジェクト切り替え時のリンク管理）ため同一 plan で扱う候補。

### F-02: コンテンツルートモジュール解決リンクが実体の node_modules を破壊的に置き換える
**戻し先:** 新規 plan（F-01 と同一 plan 候補）
**出どころ:** 利用インタビュー + コード確認（native/Sources/App/AppDelegate.swift の linkAppNodeModulesIntoContentRoot が、バンドルへのリンクでない既存アイテムを無条件に removeItem してリンクを作成する）
**解釈・原因の仮説:** 配布先（Node なし Mac）ではコンテンツフォルダに node_modules が存在しない前提が成り立つため、tdd-run のスコープ内では顕在化しなかった。開発機で既存プロジェクトをコンテンツルートに選ぶという新しい利用文脈で、実体ディレクトリの削除という破壊的挙動になる。npm install で復元可能とはいえ、ユーザーのデータを黙って消す挙動はスコープ外と言って放置できる性質ではない。
**推奨される対応:** 実体 node_modules が存在する場合は削除せず、別の解決手段（NODE_PATH 等の env による解決、リンクを張らずに起動、または警告して中断）を設計する。F-01 の「開発機での複数プロジェクト利用」という同じ問題意識に属する。

### F-03: エラー発生時の情報がユーザーに一切届かない（真っ白画面）
**戻し先:** 新規 plan
**出どころ:** 利用インタビュー
**解釈・原因の仮説:** サーバープロセスの stderr/stdout はどこにも表示・保存されておらず、起動失敗や実行時エラーが「真っ白の WKWebView」としてしか観測できない。「サーバーを起動中です」から真っ白／最初から真っ白という複数パターンがあること自体、失敗の段階（起動前・起動後）が区別できていないことを示す。native-mac-shell の起動待ち表示は正常系のみを扱っており、異常系の表示は元々どのプランのスコープにも入っていなかった。
**推奨される対応:** エラー表示（起動失敗の理由提示、サーバーログの保存先または表示手段）を対象とする新規 plan。

### F-04: 自己完結性（US-02）とGatekeeper警告のクリーン環境検証が未実施のまま
**戻し先:** 次フェーズ（配布パイプライン plan）
**出どころ:** 利用インタビュー + tests/acceptance/native-shell-distribution.spec.ts の skip 記録
**解釈・原因の仮説:** 問題定義時点から次フェーズ（GitHub Actions ビルド・zip・Releases 配布）に先送りされていた項目で、今回の利用でも検証機会がなかった。未達ではなく未検証。
**推奨される対応:** 配布パイプラインの plan を立てるときに、クリーン環境での起動検証と quarantine 属性つき .app の Gatekeeper 挙動確認を受け入れ条件に含める。

### F-05: ビルド手順が利用者に定着していない
**戻し先:** 次フェーズ（配布パイプライン plan）に吸収
**出どころ:** 利用インタビュー（「もうビルドの仕方を忘れましたが」）
**解釈・原因の仮説:** build-local.js は検証用のローカルビルドであり、ビルドが手作業・記憶頼みである状態は配布パイプライン未整備の裏返し。
**推奨される対応:** 配布パイプライン plan（CI ビルド）で自然に解消される見込み。単独の plan は不要。

### F-06: docs/dictionary.json の旧世代エントリ46件が現行フォーマット検証を通らない
**戻し先:** tdd-skills（または /tdd-vocab migrate 相当の整備作業）
**出どころ:** 成果物レビュー（promote 後の dict-write.js check で発覚）
**解釈・原因の仮説:** `en`（PascalCase 英語名）が必須化される前に登録された既存エントリが残っており、docs 全体の check が常に exit 1 になる。今回の昇格分とは無関係だが、以後のプランでも check のノイズになる。
**推奨される対応:** 旧エントリへの en 付与を一括で行う整備作業。語彙の意味は変わらないため機械的に実施可能。
