# 問題定義: 配布物が同梱コードの依存パッケージを解決できない

**バージョン:** v1
**最終更新:** 2026-07-17
**コンテキスト:** app-bundle
**関連設計資料:** scripts/app-bundle/bundledModuleResolver.js, native/Sources/App/AppDelegate.swift, lib/dir.js, docs/dictionary.json（app-bundle コンテキスト）
**性格:** 現時点の問題理解のスナップショット。design/run/feedback を経て改版されることが正常な軌道。

## 変更履歴

### v1 (2026-07-17)
- 初版。配布物（`.app`）でエディタの画像メタデータ読み取り機能を使うと `Cannot find package 'sharp'` でサーバーが起動時にクラッシュする現象から出発。コンテンツルートの実体 `node_modules` を削除しても解消しないことを確認し、原因が「コンテンツルート側の node_modules の状態」ではなく「同梱コードの依存パッケージへの参照を解決する経路がそもそも存在しない」ことにあると特定した

---

## 問題

配布物（`.app`）は「配布先環境への追加インストールを不要にする」ことを前提に、起動に必要な Node 本体・sharp のネイティブバイナリ・ソースコード一式を同梱している。しかし実際には、同梱コードの一部（コンテンツルートへ複製されるサーバーコードを含む）が使う依存パッケージへの参照が、コンテンツルート側からは解決できず、機能の利用時にサーバーがクラッシュする。

この解決の失敗は、コンテンツルートに実体の `node_modules` があるかどうか、その中身が何かに関わらず起こる。つまりコンテンツルート側の状態の問題ではなく、同梱コードの依存パッケージへの参照を同梱コード自身の実体へ導く経路が欠けているという問題である。

### やりたいこと

- 配布物（`.app`）を使う際、コンテンツルートの状態（実体 `node_modules` の有無や中身）に関わらず、配布先環境への追加インストールなしに、同梱されている機能一式が動作してほしい

### 障害・制約

- 同梱コードの一部は、実行時にコンテンツルート配下へ複製されて動作する。複製後の実体パスから見た依存解決は、コンテンツルートの祖先ディレクトリを遡る経路になり、同梱コード自身が置かれているディレクトリツリーには辿り着けない
- 同梱モジュール解決器は、アプリ自身のパッケージ名（`@tenjuu99/blog`）への自己参照だけを同梱コードへ解決する設計になっており、同梱コードが使う他の依存パッケージへの参照はその対象に含まれていない
- 過去に、コンテンツルート直下へ `node_modules` シンボリックリンクを配線し依存解決ごと丸ごと同梱コードへつなぐ方式が採られていたが、これは開発中プロジェクトの実体 `node_modules` を無条件に削除・置換してしまうという既存物保護に反する挙動を持っていたため撤去された。その撤去の副作用として、自己参照以外の依存解決経路も失われた
- 既存物保護（コンテンツルートに一切書き込まない）という既に成立している保証を崩さずに、依存解決を成立させる必要がある

### 観察された症状

- コンテンツルートに実体 `node_modules` がある状態でも、削除した状態でも、エディタの画像メタデータ読み取り機能を使うと `Error [ERR_MODULE_NOT_FOUND]: Cannot find package 'sharp'` でサーバーが起動時にクラッシュする

## 範囲外

- 既存物保護そのものの設計見直し（維持する前提とする）
- 複数OS・複数アーキテクチャ向けの sharp ネイティブバイナリの選定（配布物の組み立て時に既に解決済みの範囲）
- コンテンツルート側が自前で依存パッケージをインストールして動かす運用（今回解消したい前提そのものであり解決策ではない）

## 解決したと言える状態（問題領域の言葉で）

- 配布物としてのアプリが、コンテンツルートの状態（実体 `node_modules` の有無・中身）によらず、追加インストールなしに動作する
- 同梱コードが使う依存パッケージへの参照が、コンテンツルート配下へ複製された実行位置からであっても、常に同梱コード自身の実体へ解決される
- 既存物保護の保証（コンテンツルートに一切書き込まない）を崩さずに成立する

## 技術的背景（調査結果）

- `lib/dir.js` の `cache()` が、`packages/editor/server/` 等の同梱コードをビルド時にコンテンツルート配下の `.cache/server/` へ物理的に複製する。複製されたファイルの実体パスはコンテンツルート側になるため、そこからの bare specifier 解決はコンテンツルートの祖先ディレクトリしか辿らない
- `scripts/app-bundle/bundledModuleResolver.js` の `redirect()` は `@tenjuu99/blog`（およびそのサブパス）への参照のみを同梱コードの実体へリダイレクトする。`sharp` 等それ以外の bare specifier は対象外で、Node の通常解決に委ねられる
- `native/Sources/App/AppDelegate.swift` には、かつて `node_modules` シンボリックリンクをコンテンツルート直下に配線する処理（`linkAppNodeModulesIntoContentRoot`）があったが、既存物保護違反（`plans/archives/node-modules-destruction/problem.md`）のため撤去され、同梱モジュール解決器のみに置き換えられた。撤去時点で「実体 `node_modules` が存在する場合にアプリの動作が実際に成立するか（同梱コードとの整合）は未検証」と申し送りされていた点が、今回顕在化した
- `sharp` は `packages/editor/server/imageMetadataReader.js` と `packages/editor/server/converters/sharp.js` の双方から参照されている
- `docs/dictionary.json` の「配布物 (DistributionBundle)」の定義には「配布先環境への追加インストールを不要にする」ことが明記されており、この性質が今回破れている

## /tdd-run への申し送り

- 既存語彙（配布物、同梱モジュール解決器、既存物保護、同居）でこの問題は十分語れた。新規語彙が必要かどうかは設計内容次第（例えば「自己参照以外の依存への解決」を指す名前が要るか）は /tdd-vocab plan の判断に委ねる
- 開いている点: 対象とする依存パッケージ名をどう特定するか（`sharp` を個別に列挙するか、同梱コード自身の `package.json` の `dependencies` を動的に参照するか等）は設計判断であり、ここでは確定しない
- 開いている点: `packages/editor/server/` 以外に、コンテンツルートへ複製される同梱コードの中で同種の問題を持つ依存参照が他にもないか（`sharp` 以外にも波及するか）は未調査
- 利用検証の基準: 作者自身が実体 `node_modules` を持つ・持たない双方のコンテンツルートで配布物を実際に起動し、画像メタデータ読み取り機能が追加インストールなしに動作することで検証できる
