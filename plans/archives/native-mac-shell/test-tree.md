# テストツリー: native-mac-shell

**作成:** 2026-07-03

## できるのツリー

ネイティブアプリシェル は 起動から編集操作までを一通り行うことができる [native-shell]
├── サーバーライフサイクル連動 は サーバープロセスの起動・終了をアプリのライフサイクルに合わせることができる [native-shell]
│   └── サーバー起動完了検知 は サーバーが接続受付可能になったことを検知できる [native-shell]
├── 起動待ち表示 は サーバー起動完了検知の結果に応じて表示画面を切り替えることができる [native-shell]
│   └── 表示状態判定器 は 検知結果から次に表示すべき画面を決定できる [native-shell]
└── アプリウィンドウ は WKWebView内で編集操作（保存・ツリー操作・プレビュー）を外部への切り替えなしに実行できる [native-shell]

## ロード可能性チェック（Swift/AppKit前提）

- `サーバー起動完了検知` / `表示状態判定器`: AppKit非依存の純粋ロジック → XCTestで単体テスト可能
- `サーバーライフサイクル連動`: `Process()` によるプロセス起動・終了。AppKit非依存だが実プロセスを扱う統合寄りのテスト → XCTestで軽量な統合テストとして検証
- `アプリウィンドウ`（WKWebView内での編集操作）: AppKit/WKWebView直結で切り出せる純粋ロジックがない → 手動検証のみ（ステップ8で実アプリを起動して確認）

## 利用仮説

- 使ったらこうなるはず:
  - アプリのアイコンをダブルクリックすると、ターミナルもブラウザも介さずに1つのウィンドウが開き、サーバー起動完了後に編集画面が表示される（US-01）
  - そのウィンドウ内だけで記事の保存・プレビュー・ファイルツリー操作が完結し、別ウィンドウや外部ブラウザへの切り替えが起きない（US-02）
  - ウィンドウを閉じる/アプリを終了すると、裏で動いていたサーバープロセスも残らず終了する（US-03）
- こうなったら外れ:
  - サーバー起動前にWKWebViewへアクセスしエラー画面やクラッシュが発生する
  - 編集操作の途中で外部ブラウザや別ウィンドウが開いてしまう
  - アプリを終了してもサーバープロセスがバックグラウンドに残り続ける（`ps`で確認できてしまう）
