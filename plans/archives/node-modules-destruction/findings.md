## 発見（解釈・原因の仮説、戻し先つき）

### F-01: 「実体の node_modules をそのまま使う」はインスタンス同一性の問題で原理的に不成立
**戻し先:** なし（設計知識として記録済み）
**出どころ:** tdd-run（実装中の調査）
ヘルパー・フックは `import { allData, config } from '@tenjuu99/blog'` でモジュールレベル状態を
サーバー本体と共有している。コンテンツルート側の実体 node_modules に解決させると
別インスタンス（`allData` は空）になり、エラーなくページが空になる。
解決先のパスだけでなく **realpath の同一性（同一モジュールインスタンス）** が動作条件。
同梱バンドルの self-link（`app/node_modules/@tenjuu99/blog → ../..`）はこの条件を満たすための
既存の仕掛けであり、同梱モジュール解決器も同じ場所へ解決することで条件を保っている。
→ この知識は docs/dictionary.json の `同梱モジュール解決器` の定義に反映済み。

### F-02: 依存グラフが scripts/ を走査せず、孤立ノードチェックが適用できない
**戻し先:** tdd-skills（/tdd-scaffold depgraph の走査対象の検討）
**出どころ:** observations.md
`.claude/tdd/depgraph-regen.sh` の走査対象は lib/packages/bin 系のみ。
`scripts/app-bundle/*.js` はグラフに現れず、かつこの装置の構成参加は
Swift（AppDelegate）と manifest.json 経由のため JS 依存グラフでは原理的に追えない。
→ 走査対象拡張、または「JS 外から構成参加するモジュール」のチェック方法の検討材料。

### F-03: docs/dictionary.json の語彙が旧方式（リンク）前提のまま【対応済み】
**戻し先:** 対応済み（本セッションの /tdd-vocab promote で処理）
**出どころ:** observations.md / 成果物レビュー
- `既存物保護` — 旧リンク挙動の現在形記述を改訂し、`#同梱モジュール解決器` への参照を張って昇格（孤立も解消）
- `コンテンツルートモジュール解決リンク` — 【廃止】として改訂（導入理由の記録は保持）、`配布物` の contains を後継装置へ差し替え

### F-04: problem.md の作業ディレクトリが旧パス【クローズにより解消】
**戻し先:** なし（プランのアーカイブにより実害なし）
**出どころ:** observations.md
`**作業ディレクトリ:** /Users/amashige/dev/tenjuu99-blog` は存在しない
（現在は `/Users/amashige/dev/private/tenjuu99-blog`）。作業レポジトリ欄が無いため実害はなかった。
Swift のビルドキャッシュ（native/.build）も旧パスを記憶しており `rm -rf .build` が必要だった点は
今後リポジトリを移動した際の再現ポイントとして記録しておく。

### F-05: 同梱モジュール解決器の適用範囲と残る限界（プロジェクト独自パッケージの扱い）
**戻し先:** plans/error-visibility（問題定義の検討材料として接続）
**出どころ:** tdd-run（実装中の調査）
フックが横取りするのは `@tenjuu99/blog`（とそのサブパス）だけで、それ以外の参照は通常の
Node 解決に委ねられる。ヘルパー等は `.cache/` 配下から import されるため、解決はコンテンツルートの
実体 `node_modules` に到達する。したがって**開発中プロジェクトが独自にインストールした
パッケージは .app 上でもそのまま動く**（旧方式では実体ごと symlink に置き換えていたため、
独自パッケージは消えて解決不能になっていた。新方式で改善されたケース）。

その上で限界が3つ残る:

1. **配布先（node_modules の無いフォルダ）で独自パッケージに依存するヘルパーはエラーになる**。
   同梱 node_modules にはアプリ自身の依存しか無く、配布先には npm も無い前提のため原理的に
   避けにくい。エラーの見せ方は plans/error-visibility の領域と重なる。
2. **ネイティブアドオンの ABI 不一致**。プロジェクト側 node_modules のコンパイル済みアドオン
   （sharp 等）は利用者の開発用 Node でビルドされたもの。同梱 Node（24.x）とメジャーバージョンが
   違うと import 時にエラーになり得る。
3. **同一インスタンス保証は `@tenjuu99/blog` のみ**。アプリ本体とコンテンツ側コードが同じ
   パッケージ（例: marked）を import すると、同梱コピーとプロジェクトのコピーの別インスタンスになる。
   ステートレスな利用なら無害だが、モジュール状態の共有を前提とするパッケージでは理屈上ずれが起きる
   （現状そのような利用は無い）。

→ 1 と 2 は「解決できない状況を利用者が認識できるか」というシナリオ2の残り香であり、
plans/error-visibility の問題定義の検討材料。
