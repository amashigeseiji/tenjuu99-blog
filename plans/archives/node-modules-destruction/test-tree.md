# テストツリー: node-modules-destruction

**作成:** 2026-07-08

## できるのツリー

```
ネイティブアプリシェルは 実体の node_modules を持つ開発中プロジェクトを、
既存物に書き込まずにコンテンツルートとして使える（同居できる） [native-shell]
├── 同梱モジュール解決器は アプリ自身のパッケージ名への参照を、コンテンツルートの
│   中身に関わらず同梱コードへ解決できる [app-bundle]
└── （手動検証のみ）ネイティブアプリシェルは コンテンツルートに書き込まずに
    サーバーを起動できる [native-shell]
```

## 責務の境界（設計上の決定）

- 解決手段は **モジュール解決フック**（Node の `module.register`）を採用。アプリが同梱 Node を
  起動するときだけ `--import` で登録スクリプトを差し込み、`@tenjuu99/blog`（サブパス含む）への
  参照を同梱コード（`Contents/Resources/app`）へ解決する。
- この選択により、現在の破壊の張本人である `linkAppNodeModulesIntoContentRoot`
  （`native/Sources/App/AppDelegate.swift`）は**削除**される。コンテンツルートへの書き込みはゼロになる。
- 「実体の node_modules をそのまま使う」案は不採用。パス解決は通っても、サーバー本体
  （同梱コード）とヘルパーが読む `@tenjuu99/blog` が**別モジュールインスタンス**になり、
  `allData` 等のモジュールレベル状態を共有できず、ページが静かに空になるため（原理的破綻）。
- `contentRoot/node_modules` 直下や `node_modules/@tenjuu99` へのリンク設置案も不採用。
  開発中プロジェクトは実体の `@tenjuu99/blog` を依存として持つのが普通で、破壊が一段深く
  なるだけであり、`npm install` とも干渉するため。
- **過去にアプリが作った `contentRoot/node_modules` の同梱宛 symlink は撤去しない**（利用者決定）。
  今後コンテンツルートに一切書き込まないことを徹底する。解決はフックが勝つため実害はない。
- インスタンス同一性の根拠: 同梱バンドルでは `app/node_modules/@tenjuu99/blog → ../..`
  （app 自身への self-link、`scripts/app-bundle/manifest.json`）により realpath が
  `Resources/app` に収束する。フックも同じ場所へ解決するため、サーバー本体と同一インスタンスになる。
- シナリオ2（動作を成立させられない場合に知らせる）は、この手段の採用により
  **失敗モード自体が消滅**する（`@tenjuu99/blog` の解決がコンテンツルートの中身に依存しなくなるため）。
- 同梱 Node は 24.14.0（`.node-version`）であり、module customization hooks は安定利用できる。

## ロード可能性チェック

- `同梱モジュール解決器`: 純粋な JS モジュール（`scripts/app-bundle/`）→ `node --test` で単体テスト可能。
  ルートの検証も、子プロセスの node に `--import` を渡して解決先を観測する形で自動テスト可能
  （AppKit 不要）。
- `ネイティブアプリシェル`（配線）: AppKit 直結（起動引数の組み立て・リンク処理の削除）→
  手動検証のみ（既存の native-shell 系プランと同じ扱い）。

## 利用仮説

- 使ったらこうなるはず:
  - 実体の node_modules を持つプロジェクトをコンテンツルートに選んでも、フォルダ内の
    既存ファイル・ディレクトリが一切変更されない（US-01 シナリオ1）
  - その状態でアプリの起動〜編集が成立する（ヘルパーの `allData` がサーバーと同一インスタンスで共有される）
  - node_modules の無いフォルダ（配布先の Mac 相当）でも従来どおり起動・編集できる（US-01 シナリオ3）
  - アプリ利用後に素の `npm run dev` を実行しても、プロジェクト自身の node_modules で従来どおり動く（同居）
- こうなったら外れ:
  - コンテンツルート選択後にフォルダ内へ新しいファイル・リンクが現れる、または既存物が消える・置き換わる
  - ページのインデックス（記事一覧等）が空になる（別インスタンス問題の再発）
  - 配布先（node_modules 無し）で bare specifier が解決できず起動に失敗する
  - 「知らせて選ばせる」が必要な状況が残っている（解決がコンテンツルートの中身に依存している）
