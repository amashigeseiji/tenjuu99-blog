# Observations: node-modules-destruction

**日時:** 2026-07-08

## 実装中の気づき

- **ツリーの組み換えが起きた**。当初案は「モジュール解決方針判定器」（既存物の状態から4値の
  配線方針を決める純粋ロジック＋同意ダイアログ）だったが、ユーザーからの
  「`@tenjuu99` の位置がアプリ内 node_modules を指すようにできないのか」という問いを起点に
  調査した結果、当初案の中核ケース「実体があり解決できるならそのまま使う」が
  **原理的に破綻している**ことが判明した（パス解決は通るが、サーバー本体とヘルパーが読む
  `@tenjuu99/blog` が別モジュールインスタンスになり、`allData` 等のモジュールレベル状態を
  共有できず、ページが静かに空になる）。最終的に module customization hooks
  （`module.register` + `--import`）による「同梱モジュール解決器」を採用し、
  破壊の張本人 `linkAppNodeModulesIntoContentRoot` そのものを削除した。
- 上記の組み換えにより、**US-01 シナリオ2（動作を成立させられない場合に知らせる）の
  失敗モード自体が消滅**した。解決がコンテンツルートの中身に依存しなくなったため。
  問題定義で開いていた「同意の形」の論点も、書き込み処理の不存在により解消
  （同意が必要な操作が残っていない）。ユーザーが選択した「知らせて選ばせる」UI は
  実装不要になった。
- **problem.md の `**作業ディレクトリ:** /Users/amashige/dev/tenjuu99-blog` は実在しないパス**
  （リポジトリが `dev/private/` 配下へ移動した後の旧パス）。tdd-run の作業ディレクトリ特定
  ルールでは「作業レポジトリ」フィールドが無い場合この値は使われないため実害はなかったが、
  Swift のビルドキャッシュ（native/.build）も旧パスを記憶しており `rm -rf .build` が必要だった。
- **依存グラフの孤立ノードチェックは判定不能**だった。`.claude/tdd/depgraph-regen.sh` の
  走査対象に `scripts/` が含まれておらず、`scripts/app-bundle/bundledModuleResolver.js` が
  グラフに現れない。この装置の構成参加は Swift 側エントリポイント（AppDelegate の
  `--import` 引数）と manifest.json 経由なので JS の依存グラフでは原理的に追えない。
  構成参加の実証は統合テスト（ルートテストの子プロセス起動）と受け入れテストで代替した。
- docs/dictionary.json の既存語彙 `既存物保護` `コンテンツルートモジュール解決リンク` の
  定義文が「リンク方式が現状の挙動」という前提で書かれており、本実装後は古い。
  `コンテンツルートモジュール解決リンク` は装置としては廃止（実装から削除）された。
  promote 時に定義の改訂・廃止の扱いが必要。
- 受け入れテスト（Playwright）とスキル既定の `npm test` はサンドボックス環境では
  実行できず、サンドボックス解除が必要だった（`sandbox-exec` が zsh の起動を拒否する）。
- 実機 GUI（メニュー・フォルダ選択ダイアログ経由）での手動検証は本セッションでは未実施。
  AppDelegate が組み立てるものと同一のコマンドライン（`--import` 登録スクリプト + bin/server、
  cwd=コンテンツルート）での起動〜編集画面応答・既存物無変更は、同梱 Node
  （.app ローカルビルド）と repo の node の両方で確認済み。
