## 発見（戻し先つき）

### F-01: ルートノード名「フロントマターテンプレートローダー」が実装に存在しなかった

**戻し先:** tdd-run スキル（手順7.5 テスト→実装の照合）

**現象:**
テストツリーのルートノード `フロントマターテンプレートローダー は ディレクトリに対応するフロントマターテンプレートをエディタに挿入できる` に対応する実装関数が作られなかった。`matchTemplate` + `buildFrontmatterString` の呼び出しは `fetchData()` の内部処理として埋め込まれ、「ローダー」という名前を持つ関数は存在しない状態で tdd-run が完了した。

ルートテストはこの状態でも green になった（内部で matchTemplate と buildFrontmatterString を直接呼んでいたため）。テストが green になったことで問題が見逃された。

**検証で分かったこと:**
tdd-run の手順7.5「テスト→実装の照合」には「`describe()` の主語名が実装の関数・モジュール・クラス名と対応しているか」という確認項目があるが、対応していない場合の指示が書かれていない。「対応しているなら語彙エントリの関係フィールドに書き足す」で終わっており、対応がない場合に実装を作るよう誘導するステップがない。

また手順4では「ルートノードをソリューションドメイン語彙登録から除外する」とあるため、ルートノードは語彙登録→ウォークスルーの経路でも捕捉されにくい。

**推奨される対応（スキルへのフィードバック）:**
手順7.5 の「テスト→実装」照合に以下を追記する:
> 対応する実装が存在しない場合は、その名前を持つ関数またはモジュールを実装に追加する。ルートノードは特に要注意——ルートのテストが green になっても、ルートノード名の関数が存在しなければ照合は通過させない。

### F-02: テストの describe に技術名（URLパス）が混入し、テストが書けなくなった

**戻し先:** tdd-run スキル（手順4 ツリー確定時・手順7.5 ウォークスルー）

**現象:**
`テンプレートレゾルバー は テンプレート設定をサーバーから取得できる` の子 describe として `/get_frontmatter_templates エンドポイントからJSON設定を取得できる` という技術名（URLパス・「エンドポイント」）が入った記述が書かれた。

この describe 名が「HTTP エンドポイントをテストする」という方向にエージェントを引っ張り、「ブラウザの `fetch` に依存するためテストできない」という誤った判断につながった。結果として `it('TODO', () => {})` が置かれたまま完了した。

実際にテストすべき振る舞いは「ブログ設定から `frontmatter_templates` を読み取れる」であり、技術名を排除した describe に直せば `getTemplates(config)` という純粋関数として容易にテストできた。

**検証で分かったこと:**
tdd-run の「できるのツリー」節には「ノードに技術名が現れたらシグナル」と書かれているが、これは原則の記述にとどまり、手順の中に「describe 文字列の技術名を検査するステップ」が存在しない。シグナルとして認識する機会はあるが、手順として強制されないため見落とされた。

**推奨される対応（スキルへのフィードバック）:**
手順4（ツリー確定）または手順7.5（ウォークスルー）に以下を明示的なチェックステップとして追記する:
> 全 describe 文字列を走査し、技術名（API・URL・ライブラリ・プロトコル・ストレージ種別等）が含まれていないかを確認する。含まれている場合は振る舞いの言葉に直してから次へ進む。

### F-03: editor.js の ESM import 不可によりロジックのコピーが存在する

**戻し先:** このスキルの次サイクル（または独立した技術負債タスク）

**現象:**
`frontmatter_template.js`（ESM）のコアロジック（`matchTemplate`・`buildFrontmatterString`・`loadFrontmatterTemplate`）が、ブラウザ向けの `editor.js`（`<script>` タグで直接読み込む非 ESM 環境）でインライン コピーされている。2箇所に同じ実装が存在する状態。

現時点では各関数が10行未満のため許容範囲だが、ロジックが複雑化した場合に同期漏れが生じるリスクがある。

**推奨される対応:**
`editor.js` を `type="module"` 化するか、バンドルステップを導入することでコピーを解消できる。着手タイミングはロジックが複雑化したときでよい。変更時は両ファイルに同じ修正を適用すること（辞書の注記にも明記済み）。
