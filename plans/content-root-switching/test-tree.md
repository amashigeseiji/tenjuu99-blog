# テストツリー: content-root-switching

**作成:** 2026-07-08

## できるのツリー

```
ネイティブアプリシェルは 利用中のプロジェクトを別のプロジェクトへ切り替えて使い続けられる [native-shell]
├── コンテンツルート解決器は 記憶済みのコンテンツルートがあっても、利用者の選びなおしに応じて
│   切り替え結果を確定できる [app-bundle]
├── （手動検証のみ）コンテンツルート切り替えメニューは 切り替え操作の入口を
│   アプリの通常操作として提示できる [native-shell]
└── （手動検証のみ）ネイティブアプリシェルは 確定した切り替え結果に従い、コンテンツルート記憶を
    書き換えてサーバーと表示を新しいプロジェクトで入れ替えられる [native-shell]
```

## 責務の境界（設計上の決定）

- 選びなおしの核は既存の `コンテンツルート解決器` の拡張として扱う。初回解決（キャンセル＝起動不能エラー）と
  選びなおし（キャンセル＝現状維持）は意味論が異なるため別関数になるが、概念としては同じ解決器の責務。
- 選びなおしの帰結は新しいデータ概念 `切り替え結果` で表す（切り替え確定／取りやめで現状維持／
  blog.json 無しで退けて現状維持、の3値）。取りやめと拒否を区別するのは、拒否時のみ利用者への
  通知が必要になるため。
- `コンテンツルート記憶` の書き換えは独立ノードにしない。前プラン（native-shell-distribution）の
  「永続化は解決器に注入される配線の一部」という決定を踏襲し、AppDelegate の配線に含める。
- サーバーの入れ替えも配線扱い。既存の `サーバーライフサイクル連動` は stop → 新インスタンスで start が
  そのまま使え、新しいコアロジックは生じない。

## ロード可能性チェック（Swift/AppKit前提）

- `コンテンツルート解決器`（選びなおし）: 選択ダイアログ・存在確認を関数注入する既存設計を踏襲し
  AppKit 非依存 → 単体テスト可能（`swift run CoreTests`）
- `コンテンツルート切り替えメニュー`: NSMenu 直結で切り出せる純粋ロジックがない → 手動検証のみ
- `ネイティブアプリシェル`（入れ替え配線）: AppKit 直結、ロード時に画面操作の副作用がある → 手動検証のみ
  （既存の native-mac-shell / native-shell-distribution と同じ扱い）
- ルートノード: AppKit 配線の合成 → 実アプリ起動による手動検証（ステップ8）

## 利用仮説

- 使ったらこうなるはず:
  - プロジェクト A で使用中に、アプリの通常操作（メニュー等）から「コンテンツフォルダの変更」を選び、
    blog.json を持つプロジェクト B を指定すると、アプリは B の内容で使える状態になる（US-01 シナリオ1）
  - 切り替え後にアプリを終了して再起動すると、ダイアログなしで B のまま立ち上がる（US-01 シナリオ2）
  - 変更操作を始めてもフォルダを選ばずにやめれば、A のまま使い続けられる（US-01 シナリオ3）
  - いずれの操作にも、永続化ストアの手動操作などアプリ外部の知識を必要としない
- こうなったら外れ:
  - 切り替え後もサーバーが A の作業場所のまま動き続け、表示だけ B になる（またはその逆）
  - 切り替えをキャンセルしたのに記憶が消え、次回起動時に選択ダイアログが出る
  - blog.json の無いフォルダを選んだときにアプリが終了してしまう（初回解決のエラー動作が混入）
  - 切り替え時に古いサーバープロセスが残り、ポート競合や多重起動が起きる
