## 問題

公開する（push）・取り込む（pull）のどちらも、実行するまで実際に何がどう変わるのかが見えない。公開する・更新するは、リモートの状態を事前確認せずに commit・push を試み、リモートが進んでいた場合は失敗して生の git エラーが返る。取り込むは対象ファイルの一覧化・分類は行うが、実際の内容の変化は提示せず、実行すると候補すべてを一括で反映する。

さらに、両側で変更がある（diverged）記事や、手元で削除済みだがリモートにはまだ残っている（deletedLocally）記事は、取り込みでは常に安全側に倒して見送られる。これらは取り込みの実行結果として現れるが、問題の所在はむしろローカル側にある: 本来 push されて解消されているべき差分が、ローカルに意図せず残り続けている状態である。この意図しない差分を、中身を見た上で解消する手段が、現状どの操作からも提供されていない。

### やりたいこと

- 公開する・更新するを実行する前に、実際に送ろうとしている内容の変化を確認したい
- 取り込むを実行する前に、リモート側の差分ファイル・ローカル側の差分ファイルそれぞれの一覧と、実際の内容の変化を確認したい
- 公開する・更新するの実行がリモートの状態と噛み合わず失敗する状況を、実行前に察知したい
- ローカルに意図せず残った差分（diverged・deletedLocally相当）について、内容を見た上で、リモートを優先して手元の差分を放棄する（合わせる）という選択をしたい

### 障害・制約

- 公開する・更新するは、リモートの状態を事前確認せずに commit・push を試みる。失敗時に返るのは生の git エラーであり、執筆者に理解できる言葉になっていない
- 取り込むは、安全に取り込めるかどうかの判定（版の連なりに基づく分類）は行うが、判定結果を実行前に提示せず、実際の内容の変化も見せない
- 実際の内容の変化（何がどう変わったか）を提示する経路が現状存在しない。差分の有無を読む手段はあるが、真偽値相当の情報にとどまる
- diverged・deletedLocally は取り込みでは常にスキップされ、理由メッセージのみが示される。中身を見る手段も、見た上で解消する手段もない
- 手元の内容を意図的に放棄・上書きする操作は、「無断で上書きしない」という既存の原則と両立させる必要があり、利用者の明示的な選択があった場合に限られる
- 差分の実際の統合（両側の内容を一本化すること）を自動で行うのは技術的に難しい。統合そのものは利用者が手動で行う前提を崩さない

### 観察された症状

- 「公開する」を押すと、リモートが進んでいた場合に失敗し、生の git エラーが表示される
- 「取り込む」を押すまで、何が反映され何が見送られるのかが分からない
- diverged・deletedLocally と判定された記事は理由文言だけが示され、実際にリモートと何がどう違うのか、事態がどれほど深刻かが分からない
- ローカルに残った意図しない差分を「リモートに合わせて捨てる」手段も、「push して解消する」ための材料（実際の中身）を見る手段もない

## 範囲外

- 差分の自動統合（マージ）。差分の可視化までを提供し、実際の統合は利用者が手動編集で行う
- 複数ファイルの一括公開。独立に解決できる別問題だが、本問題の差分可視化を前提とする
- リアルタイム共同編集・編集権ロック等（sync-operations で既に範囲外と確定済み）
- git 以外の公開手段（FTP 等）における差分・版の連なりの実現方法（gitless-publication-means の範囲）

## 解決したと言える状態（問題領域の言葉で）

- 公開する・更新するを実行する前に、実際に送ろうとしている内容の変化を確認できる
- 取り込むを実行する前に、リモート側・ローカル側それぞれの差分ファイルの一覧と、実際の内容の変化を確認できる
- 公開する・更新するの実行がリモートの状態と噛み合わず失敗する状況を、実行前に察知できる
- ローカルに意図せず残った差分について、内容を見た上で、リモートを優先して手元の差分を放棄する選択ができ、選択どおりに反映される
- 利用者が明示的に選択しない限り、手元の内容が意図せず上書きされることはない
- 差分の可視化は、公開する方向・取り込む方向のどちらの場面でも同じ理解のされ方をする

## 技術的背景（調査結果）

- 公開する・更新するの実装（`lib/publishing/gitPublicationMeans.js` の `createGitReflect`）は `git add` → `git diff --cached` で変更有無を確認 → `commit` → `push` の順に実行し、リモートの先行状態を事前確認しない。push 失敗時は git のエラーをそのまま返す
- 取り込む（`lib/publishing/pull.js`）は `listRemoteFiles()` で候補を得て、`versionLineage.classify()` で pullable/skip(reason) を判定し、pullable 分をまとめて `takeFromRemote` に渡す。判定結果は実行結果としてのみ返り、事前提示のAPIはない
- 版の連なり（`lineageOf`、`createGitLineageOf`）は手元・リモート・共通の祖先の3点の内容（blob）を比較し、same/localOnly/remoteOnly/remoteAhead/localAhead/diverged/deletedLocally を判定する。この分類自体は既に実装済みで、diverged・deletedLocally の検出そのものを新たに作る必要はない
- 実際の内容の変化を提示するには、`diffFromRemote`（現状 `'modified'|''` のみを返す）を拡張するか、別の経路でローカル・リモートそれぞれの実テキストを取得して比較する必要がある。現状に欠けている部分であり、対応方法は設計フェーズで決める
- `takeFromRemote` は任意のファイル一覧を受け取れる実装になっており、diverged な1ファイルだけを明示的に取り込む操作は、既存の実行体をそのまま使える見込み（`classify` の判定をバイパスする経路を新設すればよい可能性がある）
- エディタのサイドバーは公開ステータス（未公開/更新あり/公開済み/リモートのみ）をファイルごとのバッジとして表示するのみで、クリックしても詳細は開かない（変更内容は非表示）

## /tdd-run への申し送り

- **語彙定義の種**: 差分の可視化（公開・取り込み双方で共有される、実際の内容変化を見せる機能）。差分のリセット（ローカルに残った意図しない差分を、リモートを優先して放棄する操作。利用者の明示的選択が前提）
- **既存語彙の非対称性（新規語彙の候補）**: `取り込む`（Pull）は届ける向きと対になる独立した語彙エントリを持つが、届ける向き（push相当）は `同期` の定義文中で「届ける向き（公開する・更新する）」と説明されるのみで、それ自体を指す中立的な名前を持たない。公開する・更新するは意味的に異なる遷移（未公開→公開済み／更新あり→公開済み）であり、本問題はその区別に関係なく「届ける向きの反映を実行する前」を扱う必要があるため、この非対称性が語彙定義時に問題になりうる
- **開いている点**: 実際の内容変化（テキスト差分）をどう取得・提示するか（`diffFromRemote` の拡張か、別経路か）は設計フェーズで検討する
- **開いている点**: 「差分のリセット」（常にローカルの差分を破棄しリモートを採用する、単一方向の操作）に、公開しようとする画面・取り込もうとする画面のどちらからも手が届くようにするか、入り口を一方に限定するかは設計フェーズで検討する
- **開いている点**: 公開する・更新するの事前差分確認が、diverged 相当の状況をどう検出するか。現行の `lineageOf` は取り込む文脈向けに作られている。公開する文脈での「リモートが進んでいるか」の判定方法を確認する必要がある
- **別問題の種**: 複数ファイルの一括公開（本問題の差分可視化を前提とする）
- **利用検証の基準**: 作者自身が複数マシンでの執筆・公開・取り込みの実運用の中で、実際に diverged・deletedLocally の場面に遭遇し、そこで差分を見て解消できるかで検証できる
