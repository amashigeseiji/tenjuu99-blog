## US-01: 公開前に送信内容を確認する

**ロール:** 執筆者
**目的:** 公開する・更新するを実行する前に、実際に送ろうとしている内容の変化を確認したい
**価値:** 何がリモートに送られるか分からないまま実行するリスクを避けられる

### シナリオ1: 変更のある記事を公開しようとする

```
Given ローカルで記事を編集し、まだ公開していない変更がある
When 公開する操作を実行する前に確認する
Then 実際にリモートへ送られる内容の変化(何がどう変わったか)が見える
```

### シナリオ2: 送る変更が無い状態で確認する

```
Given ローカルとリモートの内容に差がない
When 公開する操作を実行する前に確認する
Then 送られる変更が無いことが分かる
```

## US-02: 取り込み前に差分を確認する

**ロール:** 執筆者
**目的:** 取り込むを実行する前に、リモート側・ローカル側それぞれの差分ファイルの一覧と実際の内容の変化を確認したい
**価値:** 何が取り込まれ、何が取り込まれないかを実行前に把握できる

### シナリオ1: リモートが進んでいる記事を確認する

```
Given 別のマシンで公開された記事があり、手元にはまだ反映されていない
When 取り込む操作を実行する前に確認する
Then その記事がリモート側の差分ファイルとして一覧に現れ、実際の内容の変化を見られる
```

### シナリオ2: ローカルで進んでいる・未反映の記事を確認する

```
Given 手元で編集したがまだ公開していない記事がある
When 取り込む操作を実行する前に確認する
Then その記事がローカル側の差分ファイルとして一覧に現れ、実際の内容の変化を見られる
```

## US-03: 公開が失敗する状況を事前に察知する

**ロール:** 執筆者
**目的:** 公開する・更新するの実行がリモートの状態と噛み合わず失敗する状況を、実行前に察知したい
**価値:** 実行してから生のgitエラーに遭遇するのではなく、実行前に対処の要否を判断できる

### シナリオ1: 噛み合わず失敗しうる状態を確認する

```
Given 手元の記事とリモートの記事の両方に、共通の祖先から見て変更がある(diverged相当)
When 公開する操作を実行する前に確認する
Then 実行すると失敗しうる状況であることが、実行前に分かる
```

### シナリオ2: 問題なく公開できる状態を確認する

```
Given リモートが進んでおらず、手元の変更だけがある
When 公開する操作を実行する前に確認する
Then 問題なく公開できる状況であることが、実行前に分かる
```

## US-04: ローカルに意図せず残った差分を解消する

**ロール:** 執筆者
**目的:** 両側で変更がある(diverged)・手元で削除済みだがリモートには残っている(deletedLocally)記事について、内容を見た上で、リモートを優先して手元の差分を放棄したい
**価値:** 中身が分からないまま取り込みでスキップされ続ける記事を、自分の判断で解消できる

### シナリオ1: 内容を見た上でリモートを採用する

```
Given 両側で変更がある(diverged)記事があり、取り込みでは常に見送られている
When その記事の実際の内容の変化を確認し、リモートを優先する選択をする
Then 手元の差分が放棄され、内容がリモートに合わせられる
```

### シナリオ2: 内容を見た上でローカルのまま見送り続けることを選ぶ

```
Given 両側で変更がある(diverged)記事がある
When その記事の実際の内容の変化を確認し、ローカルのまま見送ることを選ぶ
Then 手元の内容は変更されず、取り込みは引き続き見送られる
```

### シナリオ3(境界): 明示的な選択がない限り上書きされない

```
Given diverged・deletedLocally相当の記事がある
When 利用者がリモートを優先する選択を行わない
Then 手元の内容は上書きされない
```
