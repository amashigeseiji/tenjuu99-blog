## 問題

現在の保存操作はローカルファイルシステムへの書き込みのみで完結する。実際にWebサイトに変更を反映させるには、別途ターミナルで `git commit → git push` を実行する必要がある。この操作は非技術者には実行できない。

### やりたいこと

- 編集内容をサーバー（GitHub経由でデプロイされる本番環境）に公開する操作を、ブラウザの編集画面から完結させたい
- [publish] のようなボタンを1つ押すだけで、ローカルの変更がGitHubにpushされ、デプロイが開始されてほしい
- コミットメッセージは固定文言でよく、入力は不要

### 障害・制約

- 現在のエディタの「save」はローカルファイル保存のみ
- git操作（commit, push）はコマンドラインツールであり、ブラウザUIから直接呼び出す手段が存在しない
- 非技術者はターミナルの操作方法を知らない

### 観察された症状

- 「保存した」のに「Webに反映されていない」という状態が発生する
- デプロイのためだけに技術者（開発者）の介在が必要になる

## 範囲外

- GitHubのCI/CDパイプライン自体の設定・変更
- コミット履歴の確認・ロールバック操作
- ブランチ管理・プルリクエスト操作
- 競合（conflict）の解消
- ローカル開発時の誤push防止（git pre-push hookで制御する。アプリの機能として実装しない）

## 解決したと言える状態（問題領域の言葉で）

- 編集画面のボタン操作だけで、保存した内容がGitHubにpushされる
- 非技術者がターミナルを一切操作せずに、コンテンツをWebサイトに公開できる
- 測定可能な受け入れ基準は設計フェーズで定める

## 技術的背景（調査結果）

- **現在の保存フロー**: クライアント → POST /editor → `fs.writeFileSync()` でローカル書き込み → レスポンスで該当ページへリダイレクト
- **git操作**: Node.jsからはシェルコマンドとして実行可能。ただしgitの認証情報（SSH鍵、トークン等）がローカル環境に設定されていることが前提となる
- **デプロイ連携**: GitHubへのpushをトリガーにデプロイが走る想定。push先のリモートリポジトリとブランチは設計フェーズで確認する

## /tdd-run への申し送り

- 「公開」という語彙（ローカル保存とWebへの反映を区別する概念）が dictionary.md 未登録 → 新規語彙の候補
- 「デプロイ」という語彙（GitHubへのpushをトリガーにWebサイトへ変更が反映される一連の流れ）が dictionary.md 未登録 → 新規語彙の候補
- publishボタンの操作対象範囲が未確定：「直前に保存した1ファイルのみ」か「未コミットの全変更」かによって「解決したと言える状態」の意味が変わる
- git認証情報（SSH鍵・トークン等）がローカルに設定されていない場合の扱いが問題として開いている
- publish操作中・完了・失敗のフィードバックをユーザーがどう知覚するかが問題として開いている
- 実装・テスト前に `.git/hooks/pre-push` を設定して誤pushを防ぐこと（アプリ実装ではなく開発環境の準備として）
