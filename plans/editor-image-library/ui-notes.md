# UI検討ノート: 画像ライブラリ

**作成:** 2026-07-14
**性格:** 設計フェーズ（/tdd-run）への参考資料。解決領域の内容なので problem.md には含めない。
**ソース:** ユーザーからのUI要望ヒアリング（2026-07-14）

## ユーザーのUI想定

- サイドバーの既存タブ（Files / 新規作成）に「画像」タブを追加する
- 画像タブを開くと、画像一覧がディレクトリツリーで表示される
- 画像を選ぶと、右側の（現在 textarea/preview が占める）領域に画像とメタデータが表示される
- 削除UIもこの右側画面に配置する

## 実現性の評価（2026-07-14 時点のコード調査に基づく）

既存構造と整合的で、無理はない:

- サイドバーのタブ基盤（`role=tablist` / `data-tab` 切り替え）は editor.html:19-22 に既にあり、タブ追加は自然な拡張
- ツリー構築・描画（`packages/editor/js/tree.js` の buildTree / renderTreeHtml）は汎用的で、画像ツリーに転用できる（リンク先の挙動のみ差し替えが必要）
- 削除・改名時の「参照ファイル提示 → 3択」は、既存のカスタム確認ダイアログ（editor.html:79-82 の #confirmDialog、動的アクションボタン対応）がそのまま使える
- Filesツリーには既にファイルごとの公開状態表示（statusMap: new/modified/published/remote-only）の前例があり、画像ツリーにも同じ表現語彙を適用できる

## 難点と対処方針

1. **右側領域は「現在編集中の記事」のフォームである**
   textarea/preview だけを画像詳細に置き換えると、上部バー（公開する・非公開にする・削除する）が記事を対象にしたまま残り、画像を見ながら記事を削除・公開してしまう誤操作の温床になる。
   → 対処: textareaAndPreview 単体ではなく、editor-options バーを含む右ペイン全体を「画像詳細モード」に切り替える。バーには画像用の操作（削除・改名）と対象画像名を表示する。
2. **編集中コンテンツの保全**: 自動保存（autoSaveInitializer, 500msデバウンス）があるため、画像詳細への切り替えで未保存変更が失われる懸念は小さい。切り替え前に保留中のデバウンスを flush すればよい。
3. **戻り導線**: 画像詳細から記事編集に戻る手段が必要（Filesタブでファイル選択、または画像詳細に「閉じる」）。「現在のファイル」表示との整合に注意。

## 代替案の検討

- **モーダル案**: 不採用を推奨。画像の一覧・整理は「管理」作業であり、狭いビューポートで一時的に開くモーダルは向かない。モーダルが向くのは将来の「ライブラリから記事に画像を挿入する」ピッカー用途（今回の問題の範囲外）。その際もツリー・詳細表示のコンポーネントは共用できるので、今回ペインで作ることは将来の妨げにならない。
- **サイドバー内に詳細表示**: 不採用。プレビュー＋メタデータ＋操作を置くには幅が足りない。

## 公開状態との接続（このUIから見えること）

- Filesツリーの公開状態表示と対称に、画像ツリー・画像詳細にも「ローカルのみ／公開済み／リモートのみ（孤児）」を表示できる
- 画像詳細ペインには「この画像を参照している記事」の一覧（削除・改名の確認にも使う情報）と公開状態を並べる場所がある
- リモート孤児画像への「リモートから取り除く」操作は、記事の「非公開にする」と対称の操作としてこのペインに置ける（スコープをどこで切るかは problem.md の開いている問題を参照）
