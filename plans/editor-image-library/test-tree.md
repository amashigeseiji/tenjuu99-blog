# テストツリー: editor-image-library（画像ライブラリ・一覧フェーズ）

**作成:** 2026-07-15
**スコープ:** 画像ライブラリのうち「一覧・詳細確認」機能（US-01）のみ。追加・削除・改名・公開状態導出・宣言（US-02〜US-06）は別セッションで扱う。

## できるのツリー

画像ライブラリ は src/image 配下の画像を一覧でき、選択した画像の詳細を確認できる ができる [editor]
├── 画像スキャナー は src/image 配下の画像ファイルを再帰的に列挙できる ができる [editor]
├── 画像メタデータ読み取り器 は画像ファイルのサイズと解像度を読み取れる ができる [editor]
├── 画像台帳 は画像パスごとの追加日時を記録・取得できる ができる [editor]
├── アップロードエンドポイント は 画像アップロード時に画像台帳へ追加日時を記録できる ができる [editor]（既存装置の拡張）
├── 画像リストコレクター は 画像スキャナー・画像メタデータ読み取り器・画像台帳 を組み合わせて画像リストを作れる ができる [editor]
├── 画像リスト表示 は 画像リストコレクターの結果をサイドバーの「画像」タブにツリー表示できる ができる [editor]（DOM依存・自動テストなし、手動確認）
└── 画像詳細表示 は 一覧から選択した画像のプレビューとメタデータを右ペインに表示できる ができる [editor]（DOM依存・自動テストなし、手動確認）

エントリーポイント: `packages/editor/template/editor.html`（サイドバー「画像」タブ追加）+ `packages/editor/js/editor.js` → 画像ライブラリ

## 利用仮説

- 使ったらこうなるはず:
  - サイドバーで「画像」タブを開くと、`src/image` 配下の全画像がサムネイル一覧として表示される（US-01 シナリオ1）
  - 画像を選択すると、ファイル名・ファイルサイズ・解像度が表示され、エディタ経由で追加された画像は追加日時も表示される（US-01 シナリオ2）
  - エディタ導入以前からある画像（台帳に記録がない画像）は、追加日時が「不明」等、記録がないことが分かる形で表示される（US-01 シナリオ3）
  - 画像が1枚もない場合は空状態（「画像がありません」等）が表示される（US-01 シナリオ4）
- こうなったら外れ:
  - 画像を追加してもエディタ経由での追加なのに追加日時が記録・表示されない
  - `src/image` 以外の場所にある画像が一覧に混ざる、または `src/image` 配下の画像が一覧から漏れる
  - 詳細表示のたびにサーバーへ追加リクエストが飛ぶ（一覧取得時のデータで完結しない）

# テストツリー: editor-image-library（画像の反映・記事編集との共存フェーズ）

**作成:** 2026-07-15
**スコープ:** 前フェーズ（US-01）の実利用確認で見つかった2つの不具合（F-01, F-02）の解消。US-02〜US-06は引き続き別セッション。

## できるのツリー

画像ライブラリ は 画像の変化を一覧に反映し続け、記事編集画面と安全に共存できる ができる [editor]
├── 画像リスト表示 は 画像タブを開くたびに最新の状態を反映できる ができる [editor]（DOM依存・自動テストなし、受け入れテストで検証）
└── 画像詳細表示 は 記事編集画面と操作エリアを共有し、画像タブを離れると自動的に閉じる ができる [editor]（DOM依存・自動テストなし、受け入れテストで検証）

エントリーポイント: `packages/editor/js/editor.js`（initSidebarTabs, loadFileInPlace, 新規作成処理, openImageDetail/closeImageDetail）→ 画像ライブラリ

### ツリーの経緯（ユーザーレビューによる組み換え）

当初は3ノード構成（画像一覧／画像詳細表示の自動クローズ／記事操作ボタンの無効化）で実装したが、
ユーザーレビューで以下の指摘を受けて2ノードに組み換えた:

- 画像詳細表示を閉じるトリガーは「記事編集への切り替え（記事リンククリック）」ではなく、
  「画像タブを離れること（サイドバータブの切り替え）」であるべき
- 記事操作ボタンを無効化（disabled/pointer-events）するのではなく、既存の記事編集ヘッダー
  （editor-options: 左にファイルパス、右に操作ボタン）をそのまま流用し、画像詳細表示中は
  左に画像パス・右は空（画像用の操作ボタンは未実装のため）に置き換える
- 上記により、明示的な「閉じる」ボタンは不要（タブ切り替えが唯一の離脱手段）

この結果、「自動的に閉じる」と「誤操作を防ぐ」は同じ実装（editor-options の中身を画像用に
差し替える）に統合されたため、ツリー上も1ノードにまとめた。

## 利用仮説

- 使ったらこうなるはず:
  - 記事編集中に画像をアップロードしてすぐ「画像」タブを開くと、リロードなしでその画像が一覧に現れる（F-01）
  - 画像詳細を開いた状態でサイドバーの「Files」タブ等に切り替えると、記事リンクをクリックしなくても画像詳細が閉じ、記事編集画面（ヘッダー含む）に戻る（F-02）
  - 画像詳細表示中は、ヘッダー左側に画像パスが、右側には記事の操作ボタンが表示されない（誤操作の防止）
- こうなったら外れ:
  - 画像タブを開いても新しい画像が反映されない
  - タブを切り替えても画像詳細が残ったまま、または記事操作ボタンが画像詳細と同時に表示されてしまう

# テストツリー: editor-image-library（画像の削除・改名フェーズ）

**作成:** 2026-07-15
**スコープ:** US-02（削除）・US-03（改名）。参照記事の逆引き・確認ダイアログ・参照更新という共通の仕組みをまとめて設計・実装する。US-04〜US-06（追加・公開状態導出・宣言）は引き続き別セッション。

## できるのツリー

画像ライブラリ は 参照している記事を考慮して画像を削除・改名することができる [editor]
├── 記事参照コレクター は 全記事を走査してそれぞれの画像参照を収集することができる [editor]
├── 参照記事逆引き器 は 画像パスのローカル参照を記事一覧として公開状態つきで返すことができる [editor]（記事参照コレクター・公開ステータス判定器 を利用。#ローカル参照 を実現する）
├── 参照記事一覧エンドポイント は 画像パスに対するローカル参照の記事一覧を提供することができる [editor]
├── 参照更新器 は 記事の内容から指定した画像参照を除去または新しいパスに書き換えることができる [editor]（#参照更新 を実現する）
├── 画像削除エンドポイント は 参照の扱いの指定に応じて画像を安全に削除することができる [editor]
├── 画像改名エンドポイント は 重複を避けつつ参照の扱いの指定に応じて画像を安全に改名することができる [editor]
├── 画像削除UI は 画像詳細表示に削除操作を追加し、参照記事があれば確認ダイアログを経て削除を実行することができる [editor]（DOM依存・自動テストなし、手動確認）
└── 画像改名UI は 画像詳細表示に改名操作を追加し、参照記事があれば確認ダイアログを経て改名を実行することができる [editor]（DOM依存・自動テストなし、手動確認）

エントリーポイント: `packages/editor/js/imageDetailDisplay.js`（削除・改名操作の追加）→ 画像ライブラリ

## 利用仮説

- 使ったらこうなるはず:
  - 参照されていない画像は確認なしで削除・改名できる（US-02 シナリオ1, US-03 シナリオ1）
  - 参照されている画像を削除・改名しようとすると、実行前に参照記事一覧（公開済み記事の識別つき）が提示され、「そのまま」「更新（除去/書き換え）」「中止」を選べる（US-02 シナリオ2, US-03 シナリオ2）
  - 選択どおりに実行される。「そのまま」なら記事は変更されず画像だけ削除・改名される。「更新」なら参照していた記事も書き換わる。「中止」なら何も変更されない（US-02 シナリオ3〜5, US-03 シナリオ3〜5）
  - 改名で重複するファイル名を指定すると実行されず、重複している旨が伝わる（US-03 シナリオ6）
- こうなったら外れ:
  - 作成者が選択する前にMarkdownファイルが書き換わる
  - 本文の参照しか検出されず、frontmatterの参照が見落とされる
  - 参照記事一覧に公開済みかどうかの区別がない
  - 改名時に重複ファイル名のチェックが行われない
  - 削除・改名後も画像台帳（image-library.json）の記録と実体が食い違う

# テストツリー: editor-image-library（削除・改名フェーズの体験改善: F-01〜F-04）

**作成:** 2026-07-15
**スコープ:** 前フェーズ（US-02/US-03、削除・改名）の利用インタビューで見つかった4件の不具合（F-01〜F-04）の解消。F-05（画像詳細への参照記事一覧表示）は問題定義の拡張が必要なため対象外（/tdd-problem に送付済み）。

## できるのツリー

画像ライブラリ は 削除・改名操作後も記事編集画面と確認ダイアログの見た目の整合性を保つことができる [editor]
├── 記事再読み込み連携 は 参照も更新して削除・改名した後に、開いている記事を再読み込みすることができる [editor]（DOM依存・自動テストなし、手動確認）
├── 確認ダイアログ選択肢表示 は 選択肢のボタンを折り返さずに表示し、中止の選択肢を視覚的に区別することができる [editor]（DOM/CSS依存・自動テストなし、手動確認）
└── 画像操作ボタン は 記事操作ボタンと見た目のトンマナを揃えることができる [editor]（CSS依存・自動テストなし、手動確認）

エントリーポイント: `packages/editor/js/editor.js`（`initImageDelete`/`initImageRename` の呼び出し配線）→ 画像ライブラリ

### 判定ロジックを切り出さなかった経緯

F-01の対応として、当初「外部更新検知器」（のちに「記事再読み込み要否判定」に改名）という
名前で、現在開いている記事が今回の参照更新で書き換えられた記事一覧に含まれるかを判定する
専用モジュール・単体テストを用意する案を提示した。ユーザーから「`referenceHandling ===
'update'` のとき常に開いている記事を再読み込みすればよく、そのための判定ロジックは
大げさではないか」という指摘を受けた。

「常に再読み込み」の場合、無関係な記事を開いたまま別の記事から参照更新を伴う削除・改名を
行うと、無関係な記事も毎回再読み込みされる（カーソル位置・スクロール位置のリセット、
プレビュー再描画）というコストを説明したが、自動保存（500msデバウンス）により未保存分の
消失リスクはごく小さく、かつ同種のリスクは既存の「取り込み後の再読み込み」処理にも
既にあるとのことで、ユーザーは「常にリロードで構わない」と判断した。これにより判定ロジック
自体が不要になり、「記事再読み込み要否判定」ノードは削除して「記事再読み込み連携」
（DOM依存の配線のみ、手動確認）に置き換えた。

## 利用仮説

- 使ったらこうなるはず:
  - 「参照も除去/書き換えて削除・改名」を選んだ直後、削除・改名対象の画像を参照していた記事をFilesタブで開いていた場合、リロードなしで書き換え後の内容が表示される（F-01）
  - 確認ダイアログの3択ボタンが折り返さずに1行で表示され、「中止」は他の2択と見た目で区別できる（F-02, F-03）
  - 「改名する」「削除する」ボタンが記事編集側の「公開する」「非公開にする」とアウトライン・ラディウス・色・フォントの面で揃って見える（F-04）
- こうなったら外れ:
  - 「参照はそのまま」を選んだケースでも記事が意図せずリロードされてしまう（副作用が過剰）
  - 開いていない記事や無関係な記事のケースでリロードが誤発火する

# テストツリー: editor-image-library（画像詳細への参照記事一覧表示: F-05）

**作成:** 2026-07-15
**スコープ:** F-05（画像詳細のメタデータに参照記事一覧が表示されない）。

problem.mdの「解決したと言える状態」の「各画像のプレビュー・ファイル名・ファイルサイズ・
解像度が表示され…」という列挙は例示であり閉じた許可リストではない。既存の`画像ライブラリ`
辞書エントリの関係にも「references: 画像参照 — 一覧・詳細に参照情報を表示する」とあり、
この機能は元々の意図の範囲内と判断し、problem.md改訂を経ずに本セッションで実装する。

## できるのツリー

画像詳細表示 は 選択した画像の参照記事一覧を、既存の参照記事一覧エンドポイントから取得して表示できる ができる [editor]（DOM依存・自動テストなし、手動確認）

エントリーポイント: `packages/editor/js/editor.js`（`openImageDetail`）→ 画像詳細表示

### 設計変更の経緯（一覧取得時に含める方式から、選択時に取得する方式へ）

当初は「`/get_image_library` が画像ごとに参照記事一覧をあわせて返す」設計（`画像ライブラリ`
エンドポイント側で記事参照コレクター・参照記事逆引き器・公開ステータス判定器を組み合わせる）
で実装した。これはUS-01フェーズの利用仮説（「詳細表示のたびにサーバーへ追加リクエストが
飛ばない」）を維持する狙いだった。

しかし受け入れテスト（`tests/acceptance/editor-image-library.spec.ts`）をフルスイートで
実行したところ、タイムアウトによる不安定な失敗が発生した。原因は、画像ライブラリ内の
全画像 × それぞれの参照記事に対して公開ステータス判定（内部でgit操作を伴う）を一覧取得の
たびに行うようになったこと。US-01フェーズの「追加リクエストをしない」という利用仮説は、
ファイル名・サイズ・解像度のようなファイルシステム由来の軽い情報を前提にしたものであり、
参照記事の公開ステータス判定（git操作を伴う重い処理）には同じ前提が成り立たなかった。

代わりに、削除・改名フローが既に使っている `参照記事一覧エンドポイント`
（`/get_image_references`）を画像詳細を開いたタイミングで個別に呼び出す方式に変更した。
コストは「選択した1枚の画像 × その参照記事」の範囲に収まり、削除・改名の確認ダイアログが
既に許容している負荷と同等になる。新規のサーバーコードは不要で、`openImageDetail` から
既存エンドポイントを呼ぶ配線のみで実現した。

## 利用仮説

- 使ったらこうなるはず: 画像詳細を開くと、ファイル名等の既存メタデータに加えて参照記事一覧
  （公開済みかどうかの識別つき）が表示され、参照がなければ「参照なし」等が分かる形で表示される
- こうなったら外れ:
  - 画像一覧を開く・画像を切り替えるだけで参照記事の取得コストがかかる（選択した画像1件分を超える範囲まで計算してしまう）
  - 選択を素早く切り替えたときに、古い選択の参照記事一覧が後から上書き表示される（レース条件）
