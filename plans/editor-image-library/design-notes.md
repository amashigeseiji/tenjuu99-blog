# 設計ノート: 画像メタデータ台帳（image.json 構想）

**作成:** 2026-07-14
**性格:** 解決領域の設計スケッチ。/tdd-run への参考資料であり、problem.md は振る舞いのみを規定する。ここに書かれた構造・名称は設計フェーズで再検討してよい。
**ソース:** ユーザーとの設計議論（2026-07-14）

## 中心アイデア

画像ごとのメタデータを保持する台帳ファイル（仮称 `image.json`）を導入し、エディタのイベントにフックして維持する。

```json
[
  {
    "name": "src/image/post/hello/a.jpg",
    "referred_by": ["src/pages/post/hello.md"],
    "published_referred_by": ["src/pages/post/hello.md"],
    "protected": false,
    "added_at": "2026-07-14T10:00:00+09:00"
  }
]
```

### フィールドの性格: 導出 vs 宣言

- **導出フィールド**（走査すれば再計算できる）
  - `referred_by` — ローカル参照。記事の保存（自動保存含む）・削除イベントで更新。削除・改名時の確認ダイアログ（参照記事の提示）に使う
  - `published_referred_by` — 公開済み参照。**同期操作（公開・更新・非公開・取り込み）のタイミングでのみ更新**。リモートから画像を取り除く判断（GC）の**唯一の根拠**
- **宣言フィールド**（再計算できない。人間の申告）
  - `protected` — 「エディタの参照検出の外で使われている」宣言。テンプレートからの参照、外部サイトからの直リンク等。理由（`protected_reason` 等）を添えられる形が望ましい。当初案の `is_used_by_template` を一般化したもの
  - 手動追加の参照エントリ
  - `added_at` — 追加イベント時に記録（再計算不能だが宣言でもない。復元不能な記録）

**再構築**は導出フィールドのみを作り直し、宣言フィールド・記録フィールドは保存する。

### なぜ参照が2フィールド必要か

非公開判断の正はリモート側の内容である。ローカル参照だけで判断すると、「記事Bをローカル編集して参照を除去（自動保存済み・未公開）→ 別記事Aの公開をトリガーにGC → リモートのBはまだ画像を表示しており壊れる」が起こる。リモートが変わる瞬間は同期操作だけなので、`published_referred_by` はそのフックで増分維持できる。

### GC（自動非公開）の安全則

`published_referred_by` が空 **かつ** `protected` でない場合に限り、同期操作のタイミングでリモートから取り除く。疑わしきは消さない。

- `protected: false` かつ `referred_by` も空 → 「参照切れ」としてUIで可視化できる（削除候補の提示）

### イベントフック一覧

- 記事の保存（自動保存含む）・削除 → `referred_by` 更新
- 画像の追加・削除・改名 → エントリの作成・削除・付け替え、`added_at` 記録
- 公開・更新・非公開 → `published_referred_by` 更新 ＋ GC判断
- 取り込み（pull） → ローカルが外部要因で変わるため、pull後に再同期

### 整合性の範囲と回復

- エディタ操作の範囲では上記フックにより整合が保たれる
- エディタ外で `src/image/` や記事が直接操作された場合、整合性は保証しない。**再構築**（ローカル走査で `referred_by`、リモート走査で `published_referred_by` を再計算、宣言は保存）で回復する

### 置き場所

宣言フィールドは失われてはならないため、`.cache/` のような使い捨て領域ではなく、**コミット・同期される場所**（`src/` 配下等）に置く。複数マシン編集・取り込みを考えると実質必須。

## 検討済み・不採用の代替案

- **規約方式**: テンプレートが参照するサイト資産を `src/image/` の外に置く規約を敷き、`src/image/` 配下は記事参照のみとする案 → 既存サイトの資産移動を要求し後方互換を壊すため不採用。宣言（`protected`）方式で代替
- **ローカル参照一本での GC 判断** → 上記「なぜ2フィールド必要か」のとおり公開サイトを壊すため不採用

## 縮退ケース

- **参照ありだがローカルに実体なし**（「参照はそのままにして削除」を選んだ後）: リモートは画像を保持し続け、公開サイトは壊れない。ローカルのプレビューだけ壊れる。公開対象の収集は「ローカルに無いがリモートに既にある画像」を欠落エラーとして扱わないこと
- **エディタ導入以前からある画像**: `added_at` が存在しない。再構築時も空のままとし、UIは追加日時なしとして表示する

## 補助機能の候補

- テンプレートのテキスト走査でリテラル画像パスを拾い、`protected` の**宣言候補として提案**する（正はあくまで宣言。走査は近似であることを忘れない）
