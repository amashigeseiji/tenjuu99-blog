## 操作の事実

- 画像詳細で `shiba-inu.jpeg` を開き、パス入力欄に `path/to/shiba-inu.jpeg`（階層つきの配置パス）を入力して「移動する」を押した

- 移動の前後で同じ画像の詳細表示が継続し、URL は変更後のパスに合わせて付け替わった
- `.md` など画像でない拡張子への付け替えも試し、エラーになった（拒否が伝わった）

## 利用仮説の照合

- 利用仮説（test-tree.md 移動フェーズ）:
  - 別の置き場所への移動で一覧・詳細・URLが追従する（S1）→ **期待側**
  - 参照記事一覧の提示と「参照も書き換え」の選択どおりの実行（S3〜6）→ **期待側**
  - パス区切り入力が正当な操作になり「無反応に見えた」混乱が解消する → **期待側**（今回はパス区切り入力が正常に受理された）
  - 画像でない拡張子への付け替えは実行されず、受け付けられない旨が伝わる（S8 / F-02）→ **期待側**（前回「無反応に見えた」拒否が、今回はエラーとして認知された）
  - 台帳の追加日時が移動後のパスに引き継がれる → **利用では未確認**（ユニットテストで固定済み: 画像移動エンドポイントのテストに台帳引き継ぎのケースあり）
- ファイル名だけの変更（改名、S2）・重複パスの拒否（S7）は今回の利用では試していない（受け入れテストで検証済み）

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: dict-search.js に promote 作業で必要な表示が2つ欠けている
**戻し先:** tdd-skills
**出どころ:** 成果物レビュー（promote 作業中）
**解釈・原因の仮説:** (1) `-s`/`-d1` の出力に wip.status（new/redefine）が表示されず、昇格の区分判断に必要な情報が取れない。(2) plans と docs に同名エントリがあると plans が docs をシャドーするため、再定義の差分提示に必要な「docs 側の現行定義」を表示する手段がない。この2点のために辞書ファイルの直接読み出しで代替してしまった
**推奨される対応:** dict-search.js に (1) wip ステータスの表示（サマリー行へのバッジ付与等）、(2) 安定層のみを検索するオプション（`--stable-only` 等）を追加する

### F-02: redefine を wip.status: new で登録する取り違えが再発した
**戻し先:** tdd-skills
**出どころ:** 成果物レビュー
**解釈・原因の仮説:** 「画像ライブラリ」の再定義が wip.status: new で登録されていた。同型のミスは過去サイクル（2026-07-15）にも記録されており、運用注意だけでは再発を防げていない
**推奨される対応:** dict-write.js add が、docs 側に同名エントリが存在する場合に警告するか wip.status を自動的に redefine にする
