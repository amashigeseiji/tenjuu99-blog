# Findings: editor-image-library（一覧フェーズ / US-01）

**日時:** 2026-07-15
**対象:** plans/editor-image-library/（画像ライブラリ・一覧・詳細確認フェーズ）

## 操作の事実

- ブラウザで実際に以下を操作した:
  - サイドバーの「画像」タブを開いて一覧を見た
  - 一覧から画像を選択して詳細（ファイル名・サイズ・解像度・追加日時）を見た
  - 記事編集画面から新しく画像をアップロードし、画像タブでその反映を確認した
  - 台帳に記録のない古い画像（エディタ導入以前からある画像）を確認した

## 利用仮説の照合

- 利用仮説（test-tree.md）: サムネイル一覧・詳細表示・追加日時表示・「不明」表示の4シナリオ
- 実際: 表示内容自体（ファイル名・サイズ・解像度・追加日時・「不明」表示）は期待どおりだった。一方、利用仮説に含まれていなかった2つの操作性の問題が実際の利用で見つかった（後述）。「期待どおりか外れか」で言うと、**利用仮説自体が範囲を狭く取りすぎていた**（一覧・詳細の表示内容のみを仮説にしており、他画面との遷移・他操作との相互作用が仮説に含まれていなかった）。

## 印象（ユーザーの言葉）

- 「詳細パネルの表示内容は問題ありません」→ 表示される情報（ファイル名・サイズ・解像度・追加日時）自体への不満はなし
- 「『閉じる』を押さないといけないことと、Filesタブを開いて記事を開こうとしても、画像を『閉じる』を押さないと記事が表示されない、ということが不便」
- 「『公開する・非公開にする・削除する』のボタンを操作エリアとして定義し、この操作エリアに画像の操作があると良い」「明示的な『閉じる』操作がなくても、記事ページをクリックしたら記事編集UIになっていてほしい」
- 「記事編集画面で画像アップロードをして、そのまま画像タブを開いても、その手順でアップロードした画像が反映されていない。リロードすれば反映される」
- この不具合の困り度合い: 「作業の流れを止める、が近いんですが、そもそも『アップロードされていないかもしれない』という認知になり、間違った行動を促す可能性があるので問題があります」（軽微ではなく、誤った判断・行動を誘発しうる）
- 補足: エディタの外で `src/image/` に直接配置した画像がリロードなしで反映されない点は、本人も範囲外として許容している（problem.mdの範囲外定義と一致）

## 発見（解釈・原因の仮説、戻し先つき）

### F-01: アップロード直後、画像タブがリロードなしでは反映されない

**戻し先:** /tdd-run（同一 plan、次サイクル）
**出どころ:** 利用インタビュー + コード確認（packages/editor/js/editor.js:639-654）
**解釈・原因の仮説:** `initImageLibrary()` は `_imageLibraryEntries` をページ読み込み時に一度だけ取得し、以降はこのキャッシュを使い回す設計（コード内コメント: 「詳細表示のたびにサーバーへ追加リクエストが飛ばない」ことを意図した最適化）。この設計は「詳細表示のたびに再取得しない」という利用仮説の外れケースを正しく回避しているが、「一覧取得後にアップロード等で実体が変わった場合の再取得」という別の要求と区別されずに一つの仕組みに混ざっている。受け入れテスト（シナリオ2）は毎回 `gotoWithRetry` でページ遷移してから検証するため、同一ページ内でアップロード→画像タブを開く、という実際の利用パターンでは再現されず、テストで検出できなかった。
**推奨される対応:** アップロード完了時（`uploadImage` 成功時）に `_imageLibraryEntries` ないし画像タブの表示を無効化・再取得するトリガーを追加する。US-02（削除）・US-03（改名）でも同種の操作後に一覧が古いまま、という同型の問題が起きうるため、次サイクルで一覧の再取得・無効化の仕組みを一箇所にまとめておくと以降のUS実装が楽になる。

### F-02: 画像詳細表示と記事編集画面の共存UIが不便（既知の懸念が実利用で顕在化）

**戻し先:** /tdd-run（同一 plan、次サイクル）
**出どころ:** 利用インタビュー + observations.md（「画像詳細表示のUI統合は当初の想定より簡略化した」）+ コード確認（packages/editor/js/editor.js:656-669）
**解釈・原因の仮説:** 実装は `.textareaAndPreview` と `#imageDetailPanel` の表示切り替えのみに簡略化されており、`.editor-options`（公開する・非公開にする・削除する等）は画像詳細表示中も出続ける。また、ファイルツリーから記事を開く経路（`loadFileInPlace`）は `closeImageDetail()` を呼ばないため、画像詳細を開いたまま記事をクリックしても記事本文に切り替わらず、明示的に「閉じる」を押す必要がある。ui-notes.md が当初懸念していた「誤操作のリスク」「右ペイン全体を画像詳細モードに切り替える」設計と、実装の簡略化版との差が、実際の利用で摩擦として顕在化した。
**推奨される対応:** ユーザー提案（「公開する・非公開にする・削除する」を操作エリアとして定義し、画像の操作もここに含める／記事クリックで暗黙に画像詳細を閉じる）を次サイクルの設計対象にする。US-02〜US-04（削除・改名）では画像側にも操作ボタンが増えるため、共存UIの整理はそれらの前に片付けておくと手戻りが少ない。

## 成果物レビュー

**語彙（plans/editor-image-library/dictionary.json）:**
- 問題なし。今回のtdd-run発見語彙（画像スキャナー・画像メタデータ読み取り器・画像台帳・画像一覧コレクター・画像一覧表示・画像詳細表示・画像ライブラリエントリ）は、それぞれ対応する実装ファイル（`src`）を持ち、責務が1対1で明確。孤立した概念もない。
- 懸念点: F-01の原因である「一覧のキャッシュを外部の変更（アップロード等）に応じて無効化・再取得する」責務は、既存のどの概念にも属していない。現状はコード内コメントのみに存在する暗黙の設計判断。次サイクルでこの責務を実装する際、新規語彙（例: 画像一覧の再取得トリガー）として切り出すか検討する価値がある。

**ソリューション構造（test-tree.md と現在の describe() 構造の比較）:**
- 差分なし。test-tree.mdの7ノード（画像スキャナー、画像メタデータ読み取り器、画像台帳、アップロードエンドポイント、画像一覧コレクター、画像一覧表示、画像詳細表示）はすべて対応する `describe()` を持つ。組み合わせ系・DOM依存系のTODOスキップも理由が明記されており妥当。
- 一方、利用仮説（test-tree.mdの「利用仮説」節）はF-01・F-02のどちらもカバーしていなかった。表示内容の正しさは仮説にあったが、「他画面・他操作との相互作用」（アップロード直後の反映、記事編集への遷移）は仮説の外にあった。

**観察メモ（observations.md）:**
- 「画像詳細表示のUI統合は当初の想定より簡略化した」という気づきは、F-02として実利用で裏付けられた。observations.mdの時点で「次フェーズ以降でUIポリッシュとして対応する必要がある」と明記されており、想定内のトレードオフとして扱われていたことと整合する。
- スコープ縮小（US-01のみ実装、US-02〜06は次回）の判断は今回の発見と矛盾しない。ただしF-01・F-02はいずれもUS-02（削除）以降で同型の問題を悪化させうるため、次サイクルの着手前に対応することを推奨する。

**promote判断:** 語彙マッピングに問題はないため `/tdd-vocab promote` を実施してよい。F-01の「再取得トリガー」概念は今回昇格させず、次サイクルで実装が固まってから語彙化する。
