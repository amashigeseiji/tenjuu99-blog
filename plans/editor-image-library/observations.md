# Observations: editor-image-library（一覧フェーズ）

**日時:** 2026-07-15

## 実装中の気づき

- **スコープを意図的に縮小した**: problem.md / user-story.md は US-01〜US-06（一覧・追加・削除・改名・公開状態導出・宣言）の6ストーリーにまたがる大きな範囲だったが、ユーザーとの相談の結果、今回のセッションは US-01（一覧・詳細確認）のみをルートXとして実装した。US-02〜US-06 は未着手のまま残っている。次回の `/tdd-run editor-image-library` は US-02（削除）から着手する想定。ルートテストと画像ライブラリ語彙は「一覧・詳細確認」の範囲に限定して green にしてあり、problem.md 全体の解決を主張するものではない。
- **`scaffold.sh` の `editor` コンテキストは `packages/editor/js/` に固定されている**が、今回追加したノード（画像スキャナー・画像メタデータ読み取り器・画像台帳・画像一覧コレクター）は fs や sharp に依存する Node 専用コードで、`packages/editor/server/` に置くのが実態に合っていた。既存の `packages/editor/server/*.js`（画像アップロードエンドポイント等）も同様に fs 依存だが scaffold の対象外だったため、今回はこれらのファイルを scaffold.sh を使わず手動で作成した。editor コンテキストが client(js/)・server(server/) の2系統を持つプロジェクト構造に scaffold.sh のマッピングが追随できていない。
- **依存グラフ（depgraph）による孤立チェックは `packages/editor/server/*.js` 全般に構造的に適用できない**: これらのファイルはディレクトリスキャン＋動的 import で読み込まれるエンドポイントのため、静的解析ベースの depgraph からは常に「依存元なし」と判定される。既存の `publish.js` / `delete.js` も同じ結果だったため、新規コードによる孤立ではなく、このプロジェクトにおける depgraph ツールの適用範囲外（動的ロードは追跡できない）と判断した。今後同様のチェックをする際は、この既知の限界を踏まえる必要がある。
- **画像詳細表示のUI統合は当初の想定より簡略化した**: ui-notes.md では「editor-options バーを含む右ペイン全体を画像詳細モードに切り替える」想定だったが、今回は `.textareaAndPreview` と `#imageDetailPanel` の表示切り替えのみとし、`.editor-options`（公開する・非公開にする・削除する等のボタン）はそのまま表示され続ける実装にした。画像詳細を見ながら誤って記事の公開・削除ボタンを押せてしまう余地が残っており、ui-notes.md が懸念していた誤操作のリスクは未解消。次フェーズ以降でUIポリッシュとして対応する必要がある。
- **受け入れテストのシナリオ4（画像0枚の空状態）は共有フィクスチャ（`src-sample/`）では検証できなかった**: 既存の画像が多数存在するため、破壊的な削除なしには「画像が1枚もない」状態を作れない。ユニットテスト（`tests/editor/image-library.test.js` のルートテスト）側で空配列ケースを検証済みのため、受け入れテストは `test.skip` として理由を明記した。
- 5回連続失敗のような詰まりポイントはなかった。葉から順に実装し、各ノードは1〜2回の実装でグリーンになった。
