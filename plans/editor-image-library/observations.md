# Observations: editor-image-library（一覧フェーズ）

**日時:** 2026-07-15

## 実装中の気づき

- **スコープを意図的に縮小した**: problem.md / user-story.md は US-01〜US-06（一覧・追加・削除・改名・公開状態導出・宣言）の6ストーリーにまたがる大きな範囲だったが、ユーザーとの相談の結果、今回のセッションは US-01（一覧・詳細確認）のみをルートXとして実装した。US-02〜US-06 は未着手のまま残っている。次回の `/tdd-run editor-image-library` は US-02（削除）から着手する想定。ルートテストと画像ライブラリ語彙は「一覧・詳細確認」の範囲に限定して green にしてあり、problem.md 全体の解決を主張するものではない。
- **`scaffold.sh` の `editor` コンテキストは `packages/editor/js/` に固定されている**が、今回追加したノード（画像スキャナー・画像メタデータ読み取り器・画像台帳・画像一覧コレクター）は fs や sharp に依存する Node 専用コードで、`packages/editor/server/` に置くのが実態に合っていた。既存の `packages/editor/server/*.js`（画像アップロードエンドポイント等）も同様に fs 依存だが scaffold の対象外だったため、今回はこれらのファイルを scaffold.sh を使わず手動で作成した。editor コンテキストが client(js/)・server(server/) の2系統を持つプロジェクト構造に scaffold.sh のマッピングが追随できていない。
- **依存グラフ（depgraph）による孤立チェックは `packages/editor/server/*.js` 全般に構造的に適用できない**: これらのファイルはディレクトリスキャン＋動的 import で読み込まれるエンドポイントのため、静的解析ベースの depgraph からは常に「依存元なし」と判定される。既存の `publish.js` / `delete.js` も同じ結果だったため、新規コードによる孤立ではなく、このプロジェクトにおける depgraph ツールの適用範囲外（動的ロードは追跡できない）と判断した。今後同様のチェックをする際は、この既知の限界を踏まえる必要がある。
- **画像詳細表示のUI統合は当初の想定より簡略化した**: ui-notes.md では「editor-options バーを含む右ペイン全体を画像詳細モードに切り替える」想定だったが、今回は `.textareaAndPreview` と `#imageDetailPanel` の表示切り替えのみとし、`.editor-options`（公開する・非公開にする・削除する等のボタン）はそのまま表示され続ける実装にした。画像詳細を見ながら誤って記事の公開・削除ボタンを押せてしまう余地が残っており、ui-notes.md が懸念していた誤操作のリスクは未解消。次フェーズ以降でUIポリッシュとして対応する必要がある。
- **受け入れテストのシナリオ4（画像0枚の空状態）は共有フィクスチャ（`src-sample/`）では検証できなかった**: 既存の画像が多数存在するため、破壊的な削除なしには「画像が1枚もない」状態を作れない。ユニットテスト（`tests/editor/image-library.test.js` のルートテスト）側で空配列ケースを検証済みのため、受け入れテストは `test.skip` として理由を明記した。
- 5回連続失敗のような詰まりポイントはなかった。葉から順に実装し、各ノードは1〜2回の実装でグリーンになった。

## 実装中の気づき（2026-07-15 / F-01・F-02 修正サイクル）

- **スコープを再度絞った**: 前回セッションのfindings.mdはF-01・F-02の解消とUS-02（削除）着手の両方を次サイクルの候補として挙げていたが、ユーザーとの相談の結果、今回はF-01・F-02の解消のみに絞った。US-02は次回に持ち越し。
- **F-01の修正方針を「再取得トリガー」から「タブを開くたびに再取得」に変更した**: 前回findings.mdは「アップロード完了時に一覧を無効化・再取得するトリガーを追加する」という設計を推奨していたが、実装時に「画像タブをクリックするたびに `initImageLibrary()` を呼び直す」という、より単純な方式を採用した。アップロード起点の専用トリガー概念を新設せずに済み、かつ将来のUS-02（削除）・US-03（改名）でも同じ仕組みで一覧が最新化されるため、前回findings.mdが懸念していた「削除・改名でも同型の問題が起きうる」への対処を先取りできた。
- **F-02の修正は`loadFileInPlace`と新規作成処理という共通の入口に`closeImageDetail()`を追加するだけで済んだ**: サイドバーリンククリック・ブラウザ戻る/進む・新規作成の3経路がすべて`loadFileInPlace`または新規作成処理を経由することが分かっていたため、個別の経路ごとに対応する必要がなかった。記事操作ボタン（公開する・非公開にする・削除する）の誤操作防止は、個々のボタンのdisabled状態を管理する既存ロジック（`fetchPublicationStatus`由来）と衝突しないよう、CSSクラストグルによる`pointer-events: none`で実現した。
- **辞書エントリの言葉選びで複数回の往復があった**: 「画像一覧表示」「画像詳細表示」の定義更新案を提示する過程で、(1) テスト所在情報（どのテストで検証されるか）を`definition`に書いてしまった、(2) 実装メカニズム（「再取得」「無効化」等の「どうやって」）を書いてしまった、(3) 画像詳細表示の定義に記事編集側の関心事（サーバーリクエストの有無、ボタン無効化、自動的に閉じる）を書いてしまった、という3段階の指摘を受けて修正した。最終的に画像詳細表示の定義は「記事編集画面とは排他的な表示関係にある」という関係性の記述のみを追加し、F-02の安全な共存という振る舞いそのものは新規語彙として登録しないことにした。
- **新規デバイス概念は登録しなかった**: 今回の修正はすべて既存関数（`initSidebarTabs`・`loadFileInPlace`・新規作成処理・`openImageDetail`・`closeImageDetail`）への配線追加であり、新しいモジュール・ファイルは作成していない。「画像一覧表示」「画像詳細表示」の定義更新のみ`plans/editor-image-library/dictionary.json`に`wip.status: "redefine"`として記録した（`add`で新規エントリとして書いてしまい、`update`で`status`を訂正する手戻りが発生した。既存の安定層エントリを再定義する際は最初から`wip.status: "redefine"`を指定すべきだった）。
- 依存グラフの孤立チェックは、新規ファイルを作成していないため対象外（スキップ）。
- **実装完了後のユーザーレビューでUI設計が組み替わった**: 最初の実装（記事操作ボタンをCSSで無効化・記事リンククリックで閉じる）に対し、ユーザーから3点の指摘を受けた: (1) 画像詳細を閉じるトリガーは「記事リンクのクリック」ではなく「画像タブを離れること（サイドバータブ切り替え）」であるべき、(2) 記事操作ボタンを無効化するのではなく、既存の記事編集ヘッダー（editor-options: 左にファイルパス・右に操作ボタン）をそのまま流用し、画像詳細表示中は左を画像パス・右を空に置き換えるべき、(3) (1)(2)を踏まえると明示的な「閉じる」ボタンは不要。この結果、`.image-detail-header`（独自のファイル名+閉じるボタン）を廃止し、`editor-options`を画像詳細表示・記事編集で共有する構造に作り直した。ツリーも3ノードから2ノードに組み換えた（詳細は test-tree.md 参照）。
- **辞書エントリの書き直しで複数回の往復があった**: 上記のUI組み換えとは別に、語彙定義の文言についても3段階の指摘（テスト所在情報を書かない／実装メカニズムを書かない／他概念の関心事を書かない）を受けた。今回はいずれもタスク完了後の設計レビューで発覚しており、実装前のツリー確認・語彙確認の段階でもっと踏み込んで質問していれば手戻りを減らせた可能性がある。次回はツリー確認時に「このノードの振る舞いを一言で言うと？」を自問し、UIの実現方法（無効化・独自ヘッダー等）を先取りしていないか確認してから実装に入るとよい。
