# Observations: publish-abstraction

**日時:** 2026-07-09

## 実装中の気づき

- 語彙定義の対話中に「ローカル」「リモート」「原稿」「成果物」が追加で発見された（ユーザー提案）。
  既存語彙（公開ステータス）の定義文がローカル/リモートを未定義のまま使っていたことが判明し、
  概念の明示が既存定義の土台を固める形になった
- 当初 editor コンテキストへの追加を想定していたが、対話の中で「公開」自体が独立した
  境界であることが見えた（ビルドは公開フローの一工程であり、editor は起動・知覚の場にすぎない）。
  publishing コンテキストを新設した
- ソース配置は一度 packages/publishing/ に置いたが、packages/ は blog.json で有効化を選ぶ
  拡張の置き場であり、publishing は常に同梱される公開機能の土台なので性格が合わないとの
  指摘（ユーザー）で lib/publishing/ に移動した。scaffold.sh の既定（未知コンテキスト →
  lib/<context>/）とも一致する。将来の CLI からの公開などコアからの利用も見込める配置
- ツリーは Strategy 的な形（同じ「できる」を差し替え可能な実現が提供する）。具体手段が
  git 1つの現時点ではパターン語彙としての登録は見送り。FTP 等の2つ目の手段が現れたら
  パターン登録を検討する
- Pipeline パターンはユーザーストーリー段階で示唆されたが、テストツリー自体はその形を
  していなかったため不採用。「パイプライン」概念の要否も対話で検討し、流れの不変部分は
  公開する・デプロイが、可変部分は公開手段が語るため不要と判断した
- packages/editor のサーバーファイルは起動時に .cache/ へコピーされて動的 import される。
  このためパッケージ外への相対 import（../../publishing/）は実行時に壊れる。
  クロスパッケージ参照は `@tenjuu99/blog/lib/publishing/...` のパッケージ指定子が必須
  （exports フィールドなし・self-link 依存）。単体テストは通るのに実行環境で壊れる罠で、
  ステップ8の実機確認で発見した
- 受け入れテストで公開操作（commit + push）を実際に走らせるため、$TMPDIR に使い捨て
  プロジェクト＋ローカル bare origin を作るフィクスチャ方式を確立した
  （tests/acceptance/publish-abstraction.spec.ts）。GitHub 等の外部に触れず push の実挙動まで検証できる
- 手動デバッグで起動した確認用サーバーがポート 8100 に残り、受け入れテストが古いサーバーに
  接続して1回空振りした（EADDRINUSE はフィクスチャ側サーバーのログにしか出ない）

## 追加サイクル（2026-07-10、findings.md F-01 対応）

- ツリー確定前にユーザーと3往復の対話が発生した。fail-fast（起動時検証で落とす）か
  グレースフルデグレード（エディター全体は動作継続）かで、ユーザーは後者を明確に選択した。
  理由は「`publish.means` は editor パッケージだけが使う値であり、他機能を止める理由がない」
- 既存の `unknown`（参照不能）ステータス判定・フロント描画がすでに存在しており、これを流用する
  ことで「means 不正」と「git remote 未設定」という原因の異なる2つの参照不能を、新しい状態を
  増やさずに統一的に扱えた。原因ごとに個別分岐を作らなかった判断はユーザーの
  「同じ処理をすることができます」という指摘がきっかけ
- ユーザーから「フロントエンドは実際にエラーを表示できているのか」と問われ、調査したところ
  `fetchPublicationStatus`・`initSidebarContent` はどちらも `res.ok` が false のとき本文を読まずに
  黙って終了する実装だった。バックエンドだけ直しても体感は変わらないという指摘は的確で、
  当初のバックエンドのみの修正案では不十分だった
- 依存グラフの孤立ノードチェックで `公開済み状態解決器` が静的依存元0件と出たが、これは
  `@tenjuu99/blog/...` 自己パッケージ指定子 + `.cache/server` 動的 import による既知の静的解析限界
  であり、`公開手段解決器`・`publish.js` 自体も同じパターンで0件だった。孤立の誤検知に注意が必要
