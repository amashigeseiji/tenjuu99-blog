# テストツリー: publish-abstraction

**作成:** 2026-07-09

## できるのツリー

```
公開手段は公開済みの参照と公開物の反映を担い、手段を替えても公開を成立させられる [publishing]
├── git公開手段は原稿を公開物としてリモートへの反映を実現できる [publishing]
├── 公開手段解決器は構成にもとづいて使う公開手段を決められる [publishing]
└── 公開ハンドラーは公開手段を通じて記事の公開・更新ができる [editor]
```

エントリポイント: `packages/editor/server/publish.js`（POST /publish）→ X
（副次: `get_publication_status.js`, `get_sidebar.js` が X の参照側面を利用）

## 利用仮説

- 使ったらこうなるはず:
  - git 手段が構成されたプロジェクトで公開操作をすると、成功がブラウザで知覚でき、
    ステータスは公開済みになり、従来どおりサイト反映フロー（push → CI/CD）が起動する（US-01 S1/S2）
  - リモートの内容が参照できない状態で公開操作をすると、公開は実行されず
    参照できない旨のフィードバックが知覚できる（US-01 S3）
  - git ではないインメモリの公開手段でも、公開フロー全体（判定→反映→公開済み）が
    同じ定義のまま成立する（US-02: 定義が手段から独立している証明）
  - 新しい公開手段（FTP 等）の検討は「その手段が定義の各要素（参照・反映・公開物）を
    どう実現するか」だけで語れる（US-02 S3）
- こうなったら外れ:
  - 公開手段を差し替えるために公開ハンドラーや公開ステータス判定器の変更が必要になる
  - git の言葉（コミット・プッシュ・リモートリポジトリ）が publishing の語彙や
    モジュールの定義文に再侵入する
  - 既存の git 運用（tests/editor/publish.test.js の既存ケース）が壊れる
