# テストツリー: publish-abstraction

**作成:** 2026-07-09

## できるのツリー

```
公開手段は公開済みの参照と公開物の反映を担い、手段を替えても公開を成立させられる [publishing]
├── git公開手段は原稿を公開物としてリモートへの反映を実現できる [publishing]
├── 公開手段解決器は構成にもとづいて使う公開手段を決められる [publishing]
├── 公開ハンドラーは公開手段を通じて記事の公開・更新ができる [editor]
├── リモート状態解決器は公開手段が解決できなくても、参照不能な状態として提供を継続し、失敗をログに残せる [publishing]（2026-07-10 追加）
└── 公開可否判定器は公開ステータスから、公開ボタンを操作可能にすべきかを判定できる [editor]（2026-07-10 追加）
```

エントリポイント: `packages/editor/server/publish.js`（POST /publish）→ X
（副次: `get_publication_status.js`, `get_sidebar.js` が X の参照側面を利用）

## 追加サイクル（2026-07-10）: findings.md F-01 への対応

**発端:** `/tdd-feedback` の利用インタビューで、`blog.json` の `publish.means` に未登録の値（例: `"ftp"`）を
設定すると、`/get_sidebar` と `/publication-status` が `resolvePublicationMeans` の例外を
`lib/tryServer.js` の汎用 catch に握りつぶされ、利用者にはサイドバー空白・404としか見えない
状態になることが判明した（F-01）。

**採用した設計判断（ユーザーとの対話で決定）:**
- fail-fast（起動時に構成を検証してサーバーを落とす）ではなく、**エディター全体は動作を継続し、
  公開操作だけが行えない**方針を採用（`publish.means` は editor パッケージのみが使う値であり、
  他機能を止める理由がないため）
- `POST /publish` は既存の外側 catch がすでに `未知の公開手段です: ftp` という正確なエラーを
  `publishFeedback` に表示できており、この経路は変更不要と判断した
- `unknown`（参照不能）ステータスは既存の `getPublicationStatus` がすでに持っており、フロントの
  描画も対応済み（`data-status="unknown"` が出るだけで壊れない）。これを流用し、
  「means が不正」と「git remote が未設定」という原因の異なる2つの参照不能を同一の `unknown` として
  統一的に扱う
- Publishボタンは `unknown` のとき無効化し、利用者に操作不可であることを能動的に伝える
  （案1: クリック後のエラー表示ではなく、事前のグレーアウト）

## 利用仮説

- 使ったらこうなるはず:
  - git 手段が構成されたプロジェクトで公開操作をすると、成功がブラウザで知覚でき、
    ステータスは公開済みになり、従来どおりサイト反映フロー（push → CI/CD）が起動する（US-01 S1/S2）
  - リモートの内容が参照できない状態で公開操作をすると、公開は実行されず
    参照できない旨のフィードバックが知覚できる（US-01 S3）
  - git ではないインメモリの公開手段でも、公開フロー全体（判定→反映→公開済み）が
    同じ定義のまま成立する（US-02: 定義が手段から独立している証明）
  - 新しい公開手段（FTP 等）の検討は「その手段が定義の各要素（参照・反映・公開物）を
    どう実現するか」だけで語れる（US-02 S3）
- こうなったら外れ:
  - 公開手段を差し替えるために公開ハンドラーや公開ステータス判定器の変更が必要になる
  - git の言葉（コミット・プッシュ・リモートリポジトリ）が publishing の語彙や
    モジュールの定義文に再侵入する
  - 既存の git 運用（tests/editor/publish.test.js の既存ケース）が壊れる

### 追加分（2026-07-10、F-01 対応）

- 使ったらこうなるはず:
  - `publish.means` に未登録の値が設定されていても、サイドバー・公開ステータス表示を含む
    エディター全体が引き続き動作する（公開操作だけができない）
  - `publish.means` が不正なとき、サーバーログにその旨が出力される
  - 公開ステータスが参照不能（unknown）なとき、原因（means 不正 / git remote 未設定）を
    問わず、Publishボタンが無効化され利用者に伝わる
- こうなったら外れ:
  - `/get_sidebar` や `/publication-status` が依然として無応答・素の404になる
  - unknown の原因ごとに個別の分岐・表示ロジックが必要になる（原因を問わず同一に扱えていない）
