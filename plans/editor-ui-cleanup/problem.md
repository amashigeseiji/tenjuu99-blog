## 問題

エディタ画面に積み重なってきた UI 要素が、現在の実際の作業区分——「新しいファイルを作る」「書く（プレビューしながら）」「保存する」「公開する」——と対応していない。フィーチャーの追加（サイドバーのディレクトリツリー、リアルタイムプレビュー）によって代替された要素がそのまま残っており、また新規作成という操作が独立したフローとして存在していない。

### やりたいこと

- 新しいファイルを作る操作を、独立した操作として行いたい。作成時にファイル名を指定し、テンプレートを選べる状態にしたい
- 新規作成によってファイルが作られた（保存された）時点では、公開はされない
- 書いた内容が保存される操作が、書く作業の流れを中断しない形で行われてほしい
- 既存の自動化された機能（リアルタイムプレビュー）に対応して、UI が整理されている状態にしたい

### 障害・制約

- 現在の「新規作成」は、主にファイルを選択するためのセレクトボックスの先頭オプションとして埋め込まれており、独立した操作として存在していない
- セレクトボックスによるファイル選択は、サイドバーのディレクトリツリーによってすでに代替されているが、UI にそのまま残っている
- リアルタイムプレビューが導入されてプレビューは自動化されているが、手動の preview ボタンが残っている
- 保存は現状まだ手動操作のみであり、書く作業の途中で明示的にボタンを押す必要がある

### 観察された症状

- 「新規作成する」がファイル選択セレクトボックスに埋め込まれており、ファイル数が増えると探しにくく、操作規則も初見では分かりづらい（空のテキスト入力にファイル名を入力する、という規則）
- サイドバーとセレクトボックスという2つのファイルナビゲーション手段が共存しており、どちらを使えばよいか分かりにくい
- ボタン群（公開状態|preview|save|publish|戻る）の中に、自動化済みの操作と手動操作が混在しており、何をいつ押すべきかの判断が必要

## 範囲外

- ファイルの削除・移動・改名（画像ライブラリ問題が担う領域）
- publish ボタン自体の機能（editor-publish で対応済み）
- サイドバーのディレクトリツリーの表示そのもの（editor-sidebar で対応済み）
- git エラーメッセージのユーザー向け変換（优先度低、editor-publish findings F-01。本プランで対応するかは設計フェーズで判断）

## 解決したと言える状態（問題領域の言葉で）

- 新しいファイルを作る操作が、独立した操作として存在し、ファイル選択と混在していない
- 新規作成時にファイル名とテンプレートを指定できる
- 新規作成でファイルが作られた時点では公開されていない（保存のみ）
- 保存操作が書く作業の流れを中断しない
- リアルタイムプレビューがすでに自動化されていることが UI から読み取れる（手動 preview ボタンが存在しない）
- ファイルナビゲーションの手段がひとつに統一されている（セレクトボックスが不要になっている）

## 技術的背景（調査結果）

- **現在の UI 構造** (`packages/editor/template/editor.html`):
  - `inputFileName`（テキスト入力）と `selectDataFile`（セレクトボックス）がファイル選択・新規作成を兼任
  - セレクトボックスの `<option value="">新規作成</option>` が新規作成の唯一のエントリポイント
  - `preview` / `save` / `publish` がフォームのサブミットボタンとして並存
- **サイドバー**: サイドバーのディレクトリツリーがファイルナビゲーションを担う（editor-sidebar で実装済み）
- **リアルタイムプレビュー**: editor-realtime-preview で実装済み。preview ボタンは機能的に代替されている
- **フロントマターテンプレート**: `packages/editor/js/frontmatter_template.js` に実装済み。ディレクトリとテンプレートの対応が `blog.json` に設定される
